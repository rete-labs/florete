C1 introduces the **ctrl-plane artifact stream** — a parallel, separately-signed delivery channel that carries the dynamic decisions an autonomous control plane will make in B1+. C1 itself has no autonomous CP yet: in C1 the ctrl artifact is produced by `florctl compile` (operator-side), every "decision" equals the corresponding [mgmt bound](../mgmt-plane/), and the framework runs in its degenerate mode. The point of landing the framework in C1 is to lock in the **shape** — envelope, signature, distribution, agent-side verification — before the producer changes in B1+.

Read [Mgmt Plane](../mgmt-plane/) first. The bounds described there are what every ctrl decision is verified against; this page assumes you know what's signed by the operator and what's left for ctrl to fill in.

## What lives in the ctrl artifact

In C1 only **cluster-flor** has ctrl-shaped state — it's where multi-hop forwarding lives. Link-flor and the agent itself have nothing dynamic in C1, so no ctrl artifacts are emitted for them. (Link-up/down liveness will likely appear here in B1+; the slot exists, just unused.)

The C1 ctrl artifact for cluster-flor:

```json
{
  "schema_version": "1.0",
  "plane": "ctrl",
  "kind": "vertex",
  "version": 17,
  "obeys_mgmt_version": 42,
  "node": "alpha",
  "generated_at": "2026-04-20T12:00:00Z",
  "payload": {
    "active_peers": [
      "spiffe://rete-lovers/vertices/beta/cluster",
      "spiffe://rete-lovers/vertices/gamma/cluster"
    ],
    "forwarding_table": [
      { "in_label": 17, "out_peer": "spiffe://rete-lovers/vertices/beta/cluster",  "out_label": 42 },
      { "in_label": 23, "out_peer": "spiffe://rete-lovers/vertices/gamma/cluster", "out_label": 71 }
    ],
    "label_bindings": [
      {
        "scope": "ingress",
        "target_spiffe_id": "spiffe://rete-lovers/services/api",
        "principal": "spiffe://rete-lovers/users/alice",
        "local_label": 9
      },
      {
        "scope": "egress",
        "target_spiffe_id": "spiffe://rete-lovers/services/mongodb",
        "principal": "spiffe://rete-lovers/services/api",
        "local_label": 11,
        "out_label": 11
      }
    ]
  },
  "signature": {
    "alg": "ed25519",
    "key_id": "spiffe://rete-lovers/control-planes/primary",
    "value": "<base64>"
  }
}
```

Field semantics:

- **Envelope** — same shape as the mgmt envelope (see [C0's spec](../../c0-tended-tunnels/mgmt-plane/validate-and-compile#compile-step)) with two additions: `plane: "ctrl"` and `obeys_mgmt_version`. The latter pins which mgmt artifact this ctrl decision was made against; agents reject ctrl whose `obeys_mgmt_version` is older than the mgmt they currently hold (rollback-attack defense).
- **`signature`** — produced by a `control-planes/<name>` key. The `key_id` is the CP's SPIFFE ID. Agents verify the signature against the pubkey listed in mgmt's `authorized_control_planes`; ctrl signed by an unauthorized key is rejected.
- **`active_peers`** — the subset of mgmt's `allowed_peers` that this cluster-flor should currently dial / accept inbound from. In C1 this equals the full set produced by `allowed_peers`. In B1+ the CP picks based on health (drain, fail-over). This is **all** the cluster-wide topology information a per-node agent ever sees — it knows its own peers, nothing else.
- **`forwarding_table`** — MPLS-style label-forwarding rows for each transit hop this node carries. In-label uniqueness is enforced by the producer; the agent verifies that each `out_peer` is in `active_peers`.
- **`label_bindings`** — binds each accepted (initiator, target) pair from the mgmt allow lists to a local label. Each binding's (scope, target_spiffe_id, principal) **must** appear in the corresponding `ingress.allow` or `egress.allow` flat list; this is the local verifier's check.

**No `active_paths` field.** Per-node agents have no use for the cluster-wide path enumeration: a flor vertex can only act on packets it actually sees, which means its own forwarding rows and label bindings. Knowing that label 17 globally corresponds to "alice → alpha → beta → kafka" buys nothing operationally — the vertex doesn't process the path, it processes labels. Keeping path enumeration out of per-node ctrl artifacts also matches the practical limit of local verifiability (see "What hijacked CP can and cannot do" below).

## Source layout (additions to mgmt-plane's)

```
my-cluster/
├── …                            # mgmt YAMLs and compiled mgmt subtree (see mgmt-plane)
├── .gitignore                   # contains: .flor/compiled/*/ctrl/
└── .flor/
    └── compiled/
        └── alpha/
            ├── mgmt/            # signed by operator; committed
            │   └── …
            └── ctrl/            # signed by CP key; gitignored
                └── vertices/
                    └── cluster.json
```

Ctrl artifacts are gitignored even in C1, where they're deterministic from operator YAMLs. The reasoning is lifecycle, not freshness: ctrl is runtime state, not authoring state. Committing it would conflate the audit log (mgmt) with the running configuration (ctrl) and create workflow drift at C1→B1+, where ctrl genuinely doesn't belong in git.

If an operator wants to record a snapshot of ctrl state for audit (e.g., "what forwarding decisions were live during the incident on May 3?"), it goes in a separate audit log, not in the cluster repo.

## Compile step (in C1)

The same `florctl compile` invocation that produces mgmt artifacts also produces ctrl artifacts in C1. There's no separate command — they're emitted from the same source-of-truth YAMLs in one pass:

```
florctl compile --node alpha --repo my-cluster/
# emits:
#   .flor/compiled/alpha/mgmt/agent.json
#   .flor/compiled/alpha/mgmt/vertices/public.json
#   .flor/compiled/alpha/mgmt/vertices/cluster.json
#   .flor/compiled/alpha/ctrl/vertices/cluster.json
```

The mgmt artifacts are signed with the operator's mgmt signing key; the ctrl artifact is signed with the operator's CP signing key. In C0/C1 both keys live on the operator's workstation — same machine, separate files (`~/.flor/operator-mgmt.key` and `~/.flor/control-planes/primary.key`). They are *not* the same key; conflating them would couple mgmt-signing-authority lifecycle with CP-signing-authority lifecycle, which becomes painful as soon as B1+ wants to put the CP key on a different host (managed cloud or BYO on-prem).

`florctl publish` pushes both subtrees:

- Mgmt artifacts go to `POST /publish/mgmt` on the config-server.
- Ctrl artifacts go to `POST /publish/ctrl`.

Authentication for the publish side is via the existing `control-plane-write` group (operator role). Internally the config-server stores mgmt and ctrl on parallel paths; agents fetch via parallel endpoints (`GET /artifact/mgmt/<node>` and `GET /artifact/ctrl/<node>`).

## Label allocation

The label allocator is a ctrl-plane concern in B1+. In C1 it runs at compile time as a deterministic walk over operator-declared paths:

- Per-link counter; for each path, walk hop-by-hop and assign the next unused labels for forward and backward directions.
- The full label map for the cluster is implicit — each node's ctrl artifact carries only its local slice (forwarding rows where this node is the transit, label bindings where this node hosts the initiator or target).
- `florctl compile --repo X` produces identical output on any machine given identical inputs (same deterministic walk order as the rest of compile).

In B1+ the allocator runs continuously inside the CP and re-runs whenever active paths change. The wire format is the same; only the producer changes.

The validator surfaces label infeasibility (per-link label space exhausted) at `florctl compile` time in C1 — not during mgmt validation, since label feasibility is intrinsically a function of the chosen path set, not the bound.

## Agent-side verification

Each agent fetches both streams and verifies them locally before applying. The flow on every sync:

1. **Fetch mgmt** for this node from `/artifact/mgmt/<node>`.
2. **Verify mgmt signature** against the pinned operator pubkey (delivered with `ca.crt` at enrollment).
3. **Fetch ctrl** for this node from `/artifact/ctrl/<node>`.
4. **Verify ctrl signature** against one of the pubkeys listed in mgmt's `authorized_control_planes`. Reject if signed by a key not in the authorized list, or if the mgmt artifact lists no authorized CP at all.
5. **Verify `obeys_mgmt_version`** ≥ the version of the mgmt artifact just verified. Reject if ctrl is older than mgmt (the CP hasn't caught up yet — wait for the next ctrl push).
6. **Verify ctrl-vs-mgmt structural rules**:
   - Every entry in `active_peers` is in the set produced by mgmt's `allowed_peers` (dispatch on each rule's `type`; for `enum` that's literal list membership).
   - Every `forwarding_table.out_peer` is in `active_peers`.
   - Every `label_bindings` entry's (scope, target_spiffe_id, principal) appears in the corresponding flat `ingress.allow` or `egress.allow` list.
   - Every `label_bindings.local_label` is referenced exactly once across the forwarding table and label-bindings list (no collisions).
7. **Apply** — combine mgmt's identities, allow lists, and adapter config with ctrl's `active_peers`, `forwarding_table`, and `label_bindings` to derive the runtime state.

If any step fails, the agent retains its previous (last-known-good) state and surfaces the failure to the operator (status, logs).

## What hijacked CP can and cannot do — the security tradeoff

This deserves to be named explicitly because it's a real and inherent tradeoff of the bounded-CP model.

**Cannot:**

- **Grant access** — the workload-layer mTLS is end-to-end against signed mgmt; a hijacked CP can't make `kafka` accept a connection from someone whose SPIFFE ID isn't in `kafka`'s mgmt-signed `ingress.allow`.
- **Add new principals or links** — those require an operator signature; the CP can only act inside what mgmt declared.
- **Persist beyond a policy refresh** — once the operator updates `authorized_control_planes` to remove the breached CP key, agents stop accepting its ctrl artifacts.

**Can:**

- **Steer traffic through unintended hops within declared peerings.** Each agent can verify that its local forwarding rows reference *its* declared peers, and that label bindings reference *its* allowed (initiator, target) pairs. But labels are opaque at the local level — node alpha doesn't know whether label 17 came from a path that, end-to-end, traverses only operator-allowed nodes. A hijacked CP can rewire forwarding rows on intermediate nodes within their respective declared peer sets, so traffic flows through different (still-allowed) hops than the operator intended.
- **Black-hole or drop traffic** — a CP that omits forwarding rows or shrinks `active_peers` causes connections to fail. DoS class.
- **Manipulate latency or do traffic analysis** — by routing through observers along permissible paths.

**Class-level summary:** access (confidentiality, integrity, authentication) survives a hijacked CP. Availability and traffic-pattern privacy don't. For customers who can't accept that tradeoff, the **BYO on-prem CP** deployment tier (planned for B1+) keeps the CP signing key out of any cloud control. For customers who can, the managed CP tier offers the convenience of cloud-hosted bounded-CP and accepts the residual blast radius.

This tradeoff is intrinsic to "global path coherence is a non-local property and cannot be locally verified by per-node agents." Mitigations (redundant ctrl annotations the initiator's agent can fully verify; out-of-band cluster-wide consistency gossip; transparency log) are deferred until concrete pressure justifies them.

## Explicitly deferred

- Autonomous CP service. C1's ctrl producer is `florctl compile`; B1+ replaces it with a CP service.
- Multi-CP coexistence (multiple `authorized_control_planes` entries, regional CP, A/B rollouts). Schema permits it from day one; orchestration is post-C1.
- CP key rotation tooling and policy refresh on schedule.
- Hot reload of ctrl-only changes (path failover with no mgmt change). Today every ctrl change still goes through `flor agent sync`'s restart-with-commit-timeout path, like mgmt.
- Path-coherence enforcement beyond local rules — see "What hijacked CP can do" above.
- Bounds-language growth beyond `allow-list`. The typed-rule shell is in place; richer rules land when concrete features need them.

### Why land the ctrl-plane stream in C1 even with no autonomous CP

The shape of the artifact stream — envelope, signature scheme, distribution endpoints, agent-side verification — is the public contract that every C1 deployment commits to. Pushing the split to B1+ would mean changing that contract once it's already deployed; even at pilot scale (handful of clusters) that's a flag-day migration we don't need to take.

By landing the stream in C1 with a degenerate producer (`florctl compile`) we exercise:

- the dual-stream fetch and verification logic in agents,
- the `obeys_mgmt_version` cross-reference,
- the typed-rule bounds dispatch,
- the `authorized_control_planes` issuance and rotation surface,
- the `mgmt/` + `ctrl/` directory layout in operator workflows.

All of these are forward-compatible with B1+, where only the *producer* changes (operator-machine `florctl` → CP service). No schema migration. No flag-day. The cost is moderate (extra plumbing in compile, publish, fetch, verify), but it's strictly less than the cost of changing the schema later.

### Why the CP signing identity is its own SPIFFE kind (not collapsed with config-server or operator)

Three identities show up around the cluster's coordination point, and they have different roles, lifecycles, and compromise blast radii:

- **`services/config-server`** — service identity, authenticates the *coordination service* to its clients. Compromise = clients dial an imposter; mTLS catches it.
- **`nodes/<node>`** — service-class identity for control-plane *clients* (the agent), authenticates them to config-server. Compromise = imposter agent reads its own per-node artifacts.
- **`control-planes/<name>`** — signing authority for ctrl artifacts. Compromise = forged ctrl decisions within mgmt bounds.

Conflating the CP signing identity with config-server's service identity would couple two different blast radii (a config-server breach would also forge ctrl artifacts) and two different rotation cadences (CP key rotates with operator policy; service identity rotates with regular cert hygiene). Keeping them as separate SPIFFE kinds keeps the failure modes independent.

Separating CP from operator identity matters too: in C0/C1 they happen to live on the same machine, but in B1+ the CP key moves to a different host (managed cloud or BYO on-prem). Treating them as one key from day one would create an awkward "the operator's key now lives in the cloud" moment. Treating them as separate identities means the migration is just "the CP key was always its own thing; it now happens to run elsewhere."

### Why ctrl signs even though it's bounds-checked

A bounded ctrl stream could in principle be unsigned — agents already verify every decision against the signed mgmt bounds, and unverified decisions outside bounds are rejected anyway. So why add a second signature?

Three reasons make it worth it:

- **Authenticity of the producer.** Bounds-checking only verifies that decisions are *permissible*; it doesn't verify *who* made them. A signature on ctrl tells the agent "this came from a CP the operator authorized," not just "this happens to satisfy mgmt." Distribution-path attacks (compromised config-server, MITM on a fetch) can substitute a permissible-but-unintended ctrl decision; signing catches that.
- **Clean revocation primitive.** Operator removes a CP's pubkey from `authorized_control_planes` → agents stop accepting that CP's ctrl artifacts. No code change, no data flush, no out-of-band coordination. This is the same revocation pattern as identity revocation (remove from mgmt, propagate on next sync).
- **Multi-CP coexistence.** As soon as B1+ wants regional CPs, A/B-tested CPs, or quorum CPs, each needs its own identity and its own pubkey in mgmt's authorized list. Without per-CP signatures the agent has no way to tell which CP made which decision.

The cost is ~zero — one signature verification per fetched ctrl, against a key the agent already pinned via mgmt. The security benefit is real even when the signing key is hot in B1+: forging ctrl still requires either the CP key itself or operator-policy authorization, both of which the bounded-autonomy model already accounts for.

### Why ctrl artifacts don't go in git, even in C1 where they're deterministic

In C1 ctrl is a deterministic function of operator YAMLs — same input, same output. Tempting to commit it for completeness alongside mgmt.

But ctrl's *role* is runtime state, not authoring state. Even when its content happens to be derivable, treating it as an authoring artifact creates two kinds of drift:

- **Workflow drift at C1→B1+.** In B1+ ctrl will be CP-produced, churning at seconds-to-minutes cadence. Operators trained to "compile and commit" against C1 will have to unlearn that and reorient when CP becomes autonomous. Cleanest if the workflow is the same throughout: operator commits mgmt, ctrl is regenerated downstream.
- **Audit confusion.** The git log for the cluster repo is the operator's *signed-decisions audit*. Mixing in CP-shaped state — even when it's derivable — dilutes that signal. A reviewer reading the diff has to figure out what was an authoring change vs. what was just a recompile of CP-shaped data.

So we treat ctrl as gitignored from day one. If an operator wants a ctrl-state snapshot for audit (incident response, change review), they capture it explicitly to a separate path — never to the cluster repo.
