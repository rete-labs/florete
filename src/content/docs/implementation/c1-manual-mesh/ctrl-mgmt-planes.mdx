This milestone inherits the control & management plane design from [C0. Tended Tunnels](../../c0-tended-tunnels/ctrl-mgmt-planes/) wholesale: same git repo as source of truth, same CA and enrollment bundles, same `flor agent sync` with commit-timeout, same naming scheme, same atomic-artifact distribution.

What C1 adds is the **mesh layer**: cluster-vertex as a new principal kind (with SPIFFE IDs of the form `spiffe://<cluster>/cluster-vertices/<node>`), declared inter-node links, explicit multi-hop paths, and a second flor vertex per node so the recursive stack from [Data Plane](../data-plane/) can run.

Two framing notes worth getting right up front, because the obvious mental model is wrong in opposite directions for the two new pieces:

**Cluster-vertex identity belongs to cluster-flor; the cert+key are *held* by link-flor.** Cluster-flor publishes itself as a service in the link layer — that's the recursive principle, exactly as the [HLD describes](../../../overview/high-level-design#service). Like any principal, cluster-flor delegates the mTLS handshake to a vertex (link-flor in this case), which holds the cert+key on its behalf. The shape is identical to C0 flor holding alice's cert+key: alice owns the identity, flor acts on her behalf. Link-flor isn't the cluster-vertex any more than C0 flor is alice.

**Link-flor is a full vertex, not a UDP forwarder.** It has a QUIC transport endpoint that terminates cluster-vertex ↔ cluster-vertex mTLS, a UDP connection manager for the wire, and a FlorIO interface where it serves cluster-flor (its only local principal in the standard topology). It just doesn't act on its own behalf — like every other vertex, it acts on behalf of the workloads it carries.

**Two link types from day one: QUIC and bare UDP.** Cluster-flor's connection manager runs multiple adapters simultaneously, picking per peer:

- **QUIC links** — full mTLS link-layer crypto, terminated by a separate **link-flor** vertex that cluster-flor reaches over FlorIO. Right answer for Internet-facing peerings (NAT traversal, metadata-resistance, authenticated transport).
- **Bare UDP links** — no link-layer crypto, implemented **directly inside cluster-flor** as a plain UDP socket. No link-flor process for these peerings. Right answer for private LAN/datacenter peerings where the underlay is already trusted and the cluster-vertex ↔ cluster-vertex QUIC overhead is wasted.

Both can coexist in the same cluster-flor. From the user's PoV the mesh looks the same — services are published once (in cluster-flor) and reachable through any combination of QUIC and UDP links along the path. The link-type choice is per-link, declared in `links.yaml`, transparent to workloads, and the multi-hop forwarding works uniformly across mixed link types.

This is also why publication should always go through cluster-flor in the standard topology: it's what gives operators a single mesh view regardless of which underlay each hop uses. Publishing in link-flor remains schema-allowed (see "mixed-vertex publication" in deferred work) but is not the default story for private-network optimisation; using a UDP link is.

## What C1 adds

- **Two link types** — QUIC (via link-flor) and bare UDP (direct in cluster-flor). Cluster-flor's connection manager runs both adapters simultaneously and picks per peer; the multi-hop forwarder works uniformly across mixed link types.
- **One or two flor vertices per node** instead of C0's one. Same `flor` binary, no `--role` flag — what makes a vertex link-shaped or cluster-shaped is its compiled config (transport_endpoint + connection_manager + which workloads it carries). A node with only UDP-link peerings runs just cluster-flor; nodes with any QUIC peerings additionally run link-flor.
- **Cluster-vertex as a new principal kind** — one per node per Florete cluster, distinct from C0's node principal. SPIFFE ID `spiffe://<cluster>/cluster-vertices/<node>`. Used for hop-by-hop link-layer mTLS.
- **`flor-agent` as a first-class principal**, distinct from any vertex. The agent owns the node identity; it delegates mTLS for control-plane traffic to a vertex like every other principal does.
- **`links.yaml`** — declares UDP peerings between cluster-vertices.
- **`paths.yaml`** — manual multi-hop route specs at the cluster layer. Labels are compiler-allocated; operators write hop sequences.
- **Three compiled artifacts per node** — one `agent.json` plus `vertices/{link,cluster}.json`. Operator YAML stays single-layered; the compiler fans out into the recursive structure.
- **Unified workloads schema** — `local_users` + `local_services` from C0 collapse into a single `workloads` list, with each workload's `io[]` channels carrying explicit directions (`inbound` / `outbound` / `bidirectional`). Same shape used by both vertex configs; cluster-vertex on link-flor and api on cluster-flor are different rows of the same table.
- **`flor vertex run`** — vertex daemon entry point. The agent supervises one or more of these per node.
- **Cluster-level FlorIO socket convention in `cluster.yaml`** — a default the compiler bakes into per-workload `io` entries.
- **Additional validator rules** — link validity, address feasibility, multi-hop path validity, label feasibility, principal-role coherence for services that appear as path initiators.

Everything else (CA, per-node bundles, revocation, `flor agent sync --commit-timeout`, config-server-fronted distribution, naming, version-stamped envelope, SOCKS5 proxy, emergency-access safety net) is identical to C0.

### cluster.yaml (additions only)

```yaml
cluster:
  name: rete-lovers                 # same as C0
  florio:                           # NEW in C1
    socket: /run/flor/io.sock       # default; the compiler emits this into workload io[] entries
  crypto:
    ca_cert: ca.crt
    signature_algorithm: ed25519
    cert_validity_days: 90
```

### nodes.yaml (additions only)

```yaml
nodes:
  alpha:                            # cluster-vertex name defaults to "alpha"
    addresses:
      - { name: public,   udp: 1.2.3.4:4433 }
      - { name: internal, udp: 10.0.0.5:4433 }
  gamma:
    addresses:
      - { name: internal, udp: 10.0.0.7:4433 }
    internal_only: true             # not reachable from Internet; only via peers
```

By convention, the cluster-vertex at node `alpha` has SPIFFE ID `spiffe://rete-lovers/cluster-vertices/alpha`. A node joining multiple Florete clusters appears in each cluster's repo under whatever name that cluster uses; cross-cluster multi-tenancy is handled at the repo level, not in `nodes.yaml`.

### links.yaml (new)

```yaml
# Undirected peerings between cluster-vertices.
# Both endpoints must be known nodes, with addresses that can reach each other.
# `type` selects the link adapter:
#   quic — mTLS over UDP, terminated by link-flor (default; right for Internet-facing)
#   udp  — bare UDP, terminated directly in cluster-flor (right for trusted private networks)
links:
  - { a: alpha, b: beta,  via: public,   type: quic }
  - { a: alpha, b: gamma, via: internal, type: udp  }
  - { a: beta,  b: gamma, via: internal, type: udp  }
```

`via` names one of the address entries on each node. The validator checks that both sides have an address with that name and that the two addresses can plausibly reach each other (public↔public, or shared internal network).

`type` defaults to `quic` and selects which connection-manager adapter cluster-flor uses for that peering. A node with at least one `quic` link runs both cluster-flor and link-flor; a node with only `udp` links runs cluster-flor alone.

### paths.yaml (new)

```yaml
# Route notation: "<initiator>|<hop1>|<hop2>|...|<target-service>"
# Initiator forms:
#   <user-name>                 — user on their own node
#   <service>@<node>            — service instance on a specific node
# Every intermediate hop is a node; last element is the target service.
# Labels are allocated by `florctl compile`, not hand-written.
paths:
  # user → service
  - alice|alpha|api
  - alice|beta|gamma|kafka
  - bob|alpha|ssh
  # service → service (api calls db and brokers)
  - api@alpha|beta|mongodb
  - api@alpha|beta|gamma|kafka
  # fallback via alpha (post-C1 feature; reserved syntax)
  # - alice|alpha|beta|mongodb  [priority: 2]
```

## CLI additions

```
# operator (florctl) — authoring
florctl compile --node <name> [--repo <dir>] [--out <dir>]
                                            # emits <out>/<node>/{agent.json, vertices/*.json}

# node (flor) — runtime
flor vertex run --config <vertex.json>      # NEW: vertex daemon
                                            # the agent supervises one or more of these per node
```

`flor agent run --config <agent.json>` keeps its C0 behaviour and CLI shape; it just supervises N vertex instances rather than one. Exactly *how* the agent launches vertex processes — direct fork, systemd unit, future local workload-runtime interface — is intentionally not pinned in C1; the runtime API stays simple (a vertex starts when given a config, and the agent makes that happen by whatever means the platform offers).

`florctl ca sign` adds `cluster-vertices` to its `--kind` enumeration (already foreshadowed in C0).

## Additional validator rules

On top of C0's:

- **Link validity** — every link is between two distinct, known nodes with declared addresses reachable from each other.
- **Address feasibility** — link pairings use addresses that can actually reach each other (public↔public, or shared internal network). Internal-only nodes can only pair with peers on the same internal network.
- **Link-type coherence** — `udp`-type links are only allowed on shared internal networks (not over Internet-facing addresses), since they carry no link-layer crypto. `quic`-type links are allowed everywhere.
- **Multi-hop path validity** —
  - Initiator is a user or `service@node`.
  - Every intermediate hop is a node.
  - Every adjacent pair of hops has a declared link.
  - The last element is a published service.
  - The service's host node is the last hop before the service name.
- **Label feasibility** — per-link label space is finite; validator ensures the allocator will succeed.
- **Principal role coherence** — services that appear as path initiators must have a `role` declared.

## Compile step additions

`florctl compile` produces **three artifacts per node** in C1, all sharing a common envelope (same as in C0):

```json
{
  "schema_version": "1.0",
  "kind": "agent",          // or "vertex"
  "version": 42,
  "node": "alpha",
  "generated_at": "2026-04-20T12:00:00Z",
  "payload": { ... }
}
```

### Workloads & IO

Both vertex configs use a unified `workloads` shape — one entry per local principal/target. Each workload has zero or more `io` channels, each with a direction from flor's PoV (matching C0 scope's "Inbound/Outbound" usage):

- `inbound` — the workload connects *into* flor here; flor originates outgoing connections on its behalf. Workload acts as a **principal**.
- `outbound` — flor connects *out* to the workload here; flor delivers incoming connections. Workload acts as a **target** (service).
- `bidirectional` — single channel does both. Used for FlorIO between recursive flor layers, where one socket carries Connect requests downward and inbound deliveries upward.

A workload entry:

```json
{
  "spiffe_id": "spiffe://rete-lovers/services/api",
  "identity": { "cert_path": "...", "priv_path": "..." },
  "io": [
    { "kind": "tcp",    "upstream": "127.0.0.1:8000",  "direction": "outbound" },
    { "kind": "socks5", "listen":   "127.0.0.1:18000", "direction": "inbound"  }
  ]
}
```

`io.kind` is open-ended; today's inventory is `socks5`, `tcp`, `florio` (the latter is new in C1). New northbound interfaces add new kinds without restructuring the schema.

### `link.json` payload

```json
{
  "ca_cert_path": "~/.flor/ca.crt",
  "transport_endpoint": { "type": "quic" },
  "connection_manager": { "adapter": "udp", "listen": "0.0.0.0:4433" },
  "workloads": [
    {
      "spiffe_id": "spiffe://rete-lovers/cluster-vertices/alpha",
      "identity":  { "cert_path": "~/.flor/cv-alpha.crt", "priv_path": "~/.flor/cv-alpha.key" },
      "io": [
        { "kind": "florio", "socket": "/run/flor/io.sock", "direction": "bidirectional" }
      ]
    }
  ],
  "ingress": [
    {
      "target_spiffe_id": "spiffe://rete-lovers/cluster-vertices/alpha",
      "allow": [
        "spiffe://rete-lovers/cluster-vertices/beta",
        "spiffe://rete-lovers/cluster-vertices/gamma"
      ]
    }
  ],
  "egress": [
    {
      "target_spiffe_id": "spiffe://rete-lovers/cluster-vertices/beta",
      "via": { "udp_peer": "5.6.7.8:4433" },
      "allow": ["spiffe://rete-lovers/cluster-vertices/alpha"]
    },
    {
      "target_spiffe_id": "spiffe://rete-lovers/cluster-vertices/gamma",
      "via": { "udp_peer": "10.0.0.7:4433" },
      "allow": ["spiffe://rete-lovers/cluster-vertices/alpha"]
    }
  ]
}
```

Things worth pointing out about this shape:

- **The cert+key are link-flor's loaded material, not its identity.** `cluster-vertices/alpha` is a workload entry that link-flor serves and acts on behalf of. From link-flor's perspective it's just a service name with a bidirectional FlorIO channel — link-flor has no idea what a cluster-vertex semantically is. The structural mirror to C0 flor holding alice's cert+key is exact.
- **No top-level `florio` block.** FlorIO sockets are properties of individual workloads' `io` entries. The cluster-level default in `cluster.yaml` only feeds the compiler.
- **`ingress`/`egress` work just like C0's.** The link layer has the same access-control shape as the cluster layer; flor doesn't distinguish which mTLS layer it's enforcing.
- **No forwarding table, no labels, no paths.** Multi-hop forwarding lives entirely in cluster-flor.

Mixed-topology case (publishing a regular service in link-flor for private-network use): the service's workload entry simply moves from `cluster.json` into `link.json`, with no schema change. The compiler decides placement based on a YAML hint per service (default = cluster).

### `cluster.json` payload

```json
{
  "ca_cert_path": "~/.flor/ca.crt",
  "transport_endpoint": { "type": "quic" },
  "connection_manager": {
    "adapters": [
      { "name": "via-link", "adapter": "florio", "socket": "/run/flor/io.sock" },
      { "name": "internal", "adapter": "udp",    "listen": "10.0.0.5:5544"     }
    ]
  },
  "peers": [
    {
      "spiffe_id": "spiffe://rete-lovers/cluster-vertices/beta",
      "via": { "adapter": "via-link" }
    },
    {
      "spiffe_id": "spiffe://rete-lovers/cluster-vertices/gamma",
      "via": { "adapter": "internal", "udp_peer": "10.0.0.7:5544" }
    }
  ],
  "workloads": [
    {
      "spiffe_id": "spiffe://rete-lovers/nodes/alpha",
      "identity":  { "cert_path": "~/.flor/alpha.crt", "priv_path": "~/.flor/alpha.key" },
      "io": [
        { "kind": "socks5", "listen": "127.0.0.1:1080", "direction": "inbound" }
      ]
    },
    {
      "spiffe_id": "spiffe://rete-lovers/services/api",
      "identity":  { "cert_path": "~/.flor/api.crt", "priv_path": "~/.flor/api.key" },
      "io": [
        { "kind": "tcp",    "upstream": "127.0.0.1:8000",  "direction": "outbound" },
        { "kind": "socks5", "listen":   "127.0.0.1:18000", "direction": "inbound"  }
      ]
    }
  ],
  "forwarding_table": [
    { "in_label": 17, "out_peer": "spiffe://rete-lovers/cluster-vertices/beta",  "out_label": 42 },
    { "in_label": 23, "out_peer": "spiffe://rete-lovers/cluster-vertices/gamma", "out_label": 71 }
  ],
  "ingress": [
    {
      "target_spiffe_id": "spiffe://rete-lovers/services/api",
      "allow": [
        { "spiffe_id": "spiffe://rete-lovers/users/alice", "local_label": 9 }
      ]
    }
  ],
  "egress": [
    {
      "target_spiffe_id": "spiffe://rete-lovers/services/mongodb",
      "via": {
        "first_hop": "spiffe://rete-lovers/cluster-vertices/beta",
        "out_label": 11
      },
      "allow": [
        { "spiffe_id": "spiffe://rete-lovers/services/api", "local_label": 11 }
      ]
    }
  ]
}
```

The agent's principal — `nodes/alpha` — appears as a workload here so the agent can dial config-server through cluster-flor like any other principal. This is the mechanism that makes the agent's bootstrap chain work: agent → cluster-flor's SOCKS5 → mTLS to config-server. It's also why the agent doesn't need to hold any cert+key itself.

`connection_manager.adapters[]` is a **list**, one entry per active link type, each with a stable `name` that the `peers[].via.adapter` field references. The example above shows alpha running both adapters: `via-link` (FlorIO down to link-flor for QUIC peerings) and `internal` (a bare UDP socket for the trusted-LAN peering with gamma). A node with only QUIC peerings would have just the `florio` adapter; a node with only UDP peerings would have just the `udp` adapter and **no link-flor at all**. Other adapter values (`wg` and similar for third-party links) are foreshadowed in [Scope > Link Protocols](../scope#link-protocols) and [Data Plane](../data-plane/), not implemented in C1.

`peers[]` maps each peer cluster-vertex's SPIFFE ID to the adapter that reaches it (and any per-peer dial info the adapter needs — for `udp` that's the peer's UDP address; for `florio` link-flor below already knows). `forwarding_table` and `egress.via.first_hop` reference peer SPIFFE IDs; the connection manager resolves "which adapter for this peer" via the `peers` map.

`forwarding_table`, `ingress.allow[].local_label`, and `egress.via` between them encode the multi-hop machinery: each accepted (initiator, target) pair binds to a forwarding-table row via its `local_label`. Egress points at the first hop (a peer cluster-vertex SPIFFE ID); the connection manager dials through whichever adapter `peers[]` assigns.

### `agent.json` payload

```json
{
  "ca_cert_path": "~/.flor/ca.crt",
  "vertices": [
    { "name": "link",    "config": "vertices/link.json"    },
    { "name": "cluster", "config": "vertices/cluster.json" }
  ],
  "control_plane": {
    "principal":     "spiffe://rete-lovers/nodes/alpha",
    "config_server": "spiffe://rete-lovers/services/config-server"
  }
}
```

The agent supervises every vertex listed under `vertices`. To reach `config-server`, the agent acts as the principal `nodes/alpha`; it finds the SOCKS5 port for that principal by scanning the referenced vertex configs for a matching workload's `inbound` channel. CA cert path and any other shared crypto material are referenced from the standard `~/.flor/` install location populated at enrollment time.

The number and shape of vertices is fully driven by `vertices`. The standard mixed-topology has two (link + cluster); a node with only `udp`-type links runs cluster-flor alone (one vertex, no link-flor); an enrollment-bootstrap config (single-vertex, direct UDP to mgmt01, just enough for the first sync) is also one. The agent doesn't care — it enumerates whatever it's told to.

### Label allocation

- Per-link counter. For each path, walk hop-by-hop and assign the next unused labels on each link for forward and backward directions.
- Allocations stored in `.flor/compiled/labels.json`. Committed alongside the per-node artifacts — the label map is part of the published cluster state.
- `florctl compile --repo X` produces identical output on any machine given identical inputs (deterministic walk order).

## Identity model additions

C1 adds one new principal kind on top of C0's users, services, and nodes:

| Kind | In C0 | In C1 | SPIFFE ID | Used for |
|---|---|---|---|---|
| Cluster vertex | No | **Yes** | `spiffe://<cluster>/cluster-vertices/<node>` | Hop-by-hop link-layer mTLS (one per node per cluster) |

A useful "owned-by / held-by" view across all four principal kinds:

| Kind | Owned by (the principal) | Cert+key held by (the vertex on its behalf) |
|---|---|---|
| node | flor-agent | cluster-flor (via the `nodes/<node>` workload entry there) |
| cluster-vertex | cluster-flor (qua principal in the link layer) | link-flor |
| user | the human (their workload) | cluster-flor |
| service | the service workload | cluster-flor (or link-flor in mixed-topology setups) |

A vertex always *holds* identity material on someone else's behalf — that's the recursive pattern from the [HLD](../../overview/high-level-design#vertex). Cluster-flor does the cluster-layer mTLS for its workloads; link-flor does the link-layer mTLS for cluster-flor (its sole standard-topology workload). No vertex is a principal at its own layer; every vertex is a principal at the layer below.

Node-identity vs. cluster-vertex-identity isn't a "should we merge them?" question any more — they're held by different vertices, owned by different principals (flor-agent vs cluster-flor), and used for different mTLS layers (control-plane vs. mesh-transit). A node joining multiple Florete clusters runs one flor-agent per cluster (one node identity each) plus the corresponding cluster-flor + link-flor stack per cluster.

Enrollment flow is unchanged: `florctl issue-bundle --node alpha --cluster <config-server-url>` packages alpha's node identity *and* alpha's cluster-vertex identity in the same per-node bundle, alongside every user/service principal hosted at alpha. The compiler knows to add a `cluster-vertices/<node>` principal for every node in C1; operators don't declare it anywhere. SPIFFE IDs are derived from the kind + the principal name + the cluster trust domain.

The bundle may also include a current `agent.json` + per-vertex configs from the latest published state — sufficient for the agent's first run with no prior sync. Subsequent `flor agent sync` calls pull whatever the cluster currently runs (which may be richer or simpler than what was bundled).

## Explicitly deferred (in addition to C0's list)

- `florctl compile --derive-paths` — declarative path derivation from topology + access. Ships post-C1 once the topology model is stable.
- Priority / fallback paths.
- 3rd-party connection-manager adapters (WireGuard, VLESS, …) and the identity-projection rules they need. Architectural placeholder in [Scope > Link Protocols](../scope#link-protocols) and [Data Plane](../data-plane/); no implementation in C1, but the multi-adapter `connection_manager.adapters[]` shape is exercised by the QUIC + UDP pair so adding a third adapter type is purely additive.
- Mixed-vertex publication (services in link-flor instead of cluster-flor). Now a corner case rather than a planned optimisation, since UDP links cover the private-network performance story without splitting publication. Schema still allows it; YAML surface and validator rules deferred indefinitely.

### Why the agent is a separate principal from the vertex

C0's single flor process did three jobs at once: it ran the data-plane vertex, it pulled state from config-server, and it owned the node-level control-plane identity. Conflating these works for one process per node but breaks the moment there's more than one — *which* vertex talks to config-server? With a recursive stack the answer ("just one of them, by config") becomes a design question; cleaner to make the agent a first-class principal that *owns* the node identity and to treat every vertex as a pure data-plane component.

The agent doesn't *hold* the node cert+key. It owns the identity (it's the principal); it delegates the actual mTLS handshake to a vertex (cluster-flor in the standard C1 topology), exactly as alice delegates her mTLS to flor in C0. The principal/vertex distinction is the same mechanism, applied to a workload that happens to live alongside its serving vertex on the same machine.

This also maps to the HLD's [Execution Plane Agent concept](../../../overview/high-level-design#execution-plane-green): the agent is the per-node supervisor of workloads, and in C1 the workloads it supervises are flor vertex processes plus (eventually) user and service workloads.

How exactly the agent launches vertex processes — direct fork, systemd unit, future local workload-runtime interface — is intentionally not pinned in C1. The runtime API stays simple: a vertex starts when given a config; the agent makes that happen by whatever means the platform offers. The shape resembles kubelet's relationship to its container runtime; the analogue here is intentional but the WRI design is deferred.

### Why cluster-flor's identity lives in link-flor's config

Cluster-flor publishes itself as a service in the link layer — it *is* the cluster-vertex principal there. By the recursive principle, that means link-flor (its serving vertex at the link layer) holds the cluster-vertex cert+key and runs the QUIC handshake with peer cluster-vertices on cluster-flor's behalf.

The shape is identical to C0 flor holding alice's cert+key: alice owns the identity, flor holds the material and acts on alice's behalf. We never confuse alice with flor; we shouldn't confuse cluster-flor with link-flor either. Both pairings are instances of the same principal/vertex relationship, just at different layers.

A consequence worth being explicit about: link-flor's config is **vocabulary-neutral**. It speaks of `workloads`, `ingress`/`egress`, SPIFFE-ID names, and FlorIO sockets — never of "cluster vertex". From link-flor's PoV, `cluster-vertices/alpha` is just a workload it serves with bidirectional FlorIO. Mixing in a regular service there (a private-network optimisation) is a one-line YAML change with no schema impact.

Publication is **static-config-only** in C1: link-flor's local services are baked into its compiled JSON, not announced dynamically over FlorIO. This is a deliberate simplification — keeps everything publishable under the signed-config model and avoids designing the Publish side of FlorIO before we need it. Connect remains the only dynamic FlorIO verb.

### Why one `workloads` list and an `io[]` direction

C0 had `local_users` (with `socks5_proxy`) and `local_services` (with `upstream_addr`, sometimes also `socks5_proxy`). The split conflated principal-vs-target with workload-kind; C1's cluster-vertex case broke it (a cluster-vertex is both, on a single FlorIO channel).

Unifying to `workloads[].io[].direction` resolves the asymmetry: a workload is a principal if it has at least one `inbound` channel, a target if it has at least one `outbound` channel, both if both. FlorIO becomes a `bidirectional` channel — one socket, both directions, no mechanism duplication. The schema also stops privileging SOCKS5 and TCP-upstream as the only northbound shapes; new `io.kind` values can be added without restructuring.

### Why three compiled artifacts per node

C1 runs an agent plus N flor vertices (standard topology: 2). Each is a separately-spawned process with its own config; bundling them into one artifact would force the agent to demultiplex and to know vertex-internal schemas, which it has no reason to. Splitting into `agent.json` + `vertices/*.json` keeps each consumer's contract minimal: the agent reads its own list and supervises; each vertex reads its own config and runs.

Operator YAML doesn't expose this fan-out. The compiler is the only thing that thinks in vertex configs; operators think in nodes, links, services, paths.

### Why two link types from C1 (and not just QUIC)

A single-link-type C1 forces a bad tradeoff. QUIC-only is right for Internet-facing peerings (NAT traversal, metadata-resistance, mTLS at the link layer) but burns CPU and adds a second crypto layer for peerings that already cross only trusted private network — datacenter racks, office LANs, on-vehicle Ethernet. UDP-only would leave the Internet-facing case unsolved.

The clean answer is to make link type a per-link choice, with both adapters present in cluster-flor's connection manager from the start. From the workload's PoV nothing changes — the mesh looks like one mesh, services are published once in cluster-flor regardless of which underlay each hop uses. From cluster-flor's PoV the multi-hop forwarder takes a peer cluster-vertex SPIFFE ID and the `peers[]` map tells it which adapter to dial through; the forwarder doesn't know or care whether the hop is QUIC-via-link-flor or bare-UDP-direct.

A consequence worth being explicit about: bare UDP links **don't need a link-flor process**. A node with only UDP-type peerings runs cluster-flor alone, with a `udp` adapter directly bound to a UDP socket. This is also why publication should always go through cluster-flor in the standard topology — using link-flor publication for "private network optimisation" was the wrong solve; the right solve is a UDP link, which keeps the recursive principle intact and the mesh story uniform.

Implementing both adapters from C1 also exercises the multi-adapter shape of `connection_manager.adapters[]` in production. Adding a third adapter (WireGuard, VLESS, …) later is then purely additive — no reshape of the schema, no migration of existing configs.

Why bare UDP and not "QUIC without crypto"? Because the simplest possible thing wins: a UDP socket is a UDP socket. Anything with QUIC framing pays per-packet header tax that's pure waste in the trusted-LAN case. Reserving "QUIC-but-cheaper" as a future option is fine; in the meantime the bare-UDP adapter is trivial to implement, trivial to reason about, and gives operators the high-performance internal-mesh story they need.

### Why multi-hop paths need MPLS-style labels (and why they're compiler-allocated)

Multi-hop forwarding needs each intermediate vertex to know, per incoming packet, where to forward it without re-doing access-control checks end-to-end. MPLS-style labels are the standard answer: a small integer attached per hop that indexes directly into a forwarding table.

Hand-written labels are the biggest footgun in manual-mode routing — two operators edit the same link and clash; a label gets reused after deletion; the forwarding table silently blackholes traffic. Compiler-allocated labels eliminate the class: operators write hop sequences (the thing they actually want), and the allocator produces a collision-free, deterministic labelling from there.

Declarative path derivation (`--derive-paths`) is deferred because operator intent for multi-hop routing (especially redundancy and preferred paths) doesn't yet have a clean declarative shape in our model. Manual paths are the minimum viable form while the topology+access model stabilises.
