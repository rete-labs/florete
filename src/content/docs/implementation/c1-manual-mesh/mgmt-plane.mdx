This milestone inherits the management plane design from [C0. Tended Tunnels](../../c0-tended-tunnels/mgmt-plane/) wholesale: same git repo as source of truth, same CA and enrollment bundles, same `flor agent sync` with commit-timeout, same naming scheme, same atomic-artifact distribution.

C1 is also where the **ctrl-plane artifact stream** appears for the first time, even though no autonomous control-plane service exists yet. In C1 the ctrl artifact is operator-produced via `florctl compile` (degenerate case — every "decision" equals the corresponding bound), but it travels in its own envelope, has its own signature, and is verified separately by each agent against bounds expressed here in mgmt. This page covers the **bounds**; ctrl mechanics and how decisions are derived from them live in [Ctrl Plane](../ctrl-plane/).

**All source YAMLs in the cluster repo author mgmt-plane bounds.** Ctrl-plane decisions are derived: in C1 by `florctl compile`, in B1+ by an autonomous CP service. The authoring surface stays mgmt-shaped throughout — operators never edit ctrl artifacts directly.

What C1 adds is the **mesh layer**: cluster-vertex as a new principal kind (with SPIFFE IDs of the form `spiffe://<cluster>/vertices/<node>/<name>`), declared inter-node links, explicit multi-hop paths, and a second flor vertex per node so the recursive stack from [Data Plane](../data-plane/) can run.

Two framing notes worth getting right up front, because the obvious mental model is wrong in opposite directions for the two new pieces:

**Cluster-vertex identity belongs to cluster-flor; the cert+key are *held* by link-flor.** Cluster-flor publishes itself as a service in the link layer — that's the recursive principle, exactly as the [HLD describes](../../../overview/high-level-design#service). Like any principal, cluster-flor delegates the mTLS handshake to a vertex (link-flor in this case), which holds the cert+key on its behalf. The shape is identical to C0 flor holding alice's cert+key: alice owns the identity, flor acts on her behalf. Link-flor isn't the cluster-vertex any more than C0 flor is alice.

**Link-flor is a full vertex, not a UDP forwarder.** It has a QUIC transport endpoint that terminates cluster-vertex ↔ cluster-vertex mTLS, a UDP connection manager for the wire, and a FlorIO interface where it serves cluster-flor (its only local principal in the standard topology). It just doesn't act on its own behalf — like every other vertex, it acts on behalf of the workloads it carries.

**Two link types from day one: QUIC and bare UDP.** Cluster-flor's connection manager runs multiple adapters simultaneously, picking per peer:

- **QUIC links** — full mTLS link-layer crypto, terminated by a separate **link-flor** vertex that cluster-flor reaches over FlorIO. Right answer for Internet-facing peerings (NAT traversal, metadata-resistance, authenticated transport).
- **Bare UDP links** — no link-layer crypto, implemented **directly inside cluster-flor** as a plain UDP socket. No link-flor process for these peerings. Right answer for private LAN/datacenter peerings where the underlay is already trusted and the cluster-vertex ↔ cluster-vertex QUIC overhead is wasted.

Both can coexist in the same cluster-flor. From the user's PoV the mesh looks the same — services are published once (in cluster-flor) and reachable through any combination of QUIC and UDP links along the path. The link-type choice is per-link, declared in `links.yaml`, transparent to workloads, and the multi-hop forwarding works uniformly across mixed link types.

This is also why publication should always go through cluster-flor in the standard topology: it's what gives operators a single mesh view regardless of which underlay each hop uses. Publishing in link-flor remains schema-allowed (see "mixed-vertex publication" in deferred work) but is not the default story for private-network optimisation; using a UDP link is.

## What C1 adds

- **Two link types** — QUIC (via link-flor) and bare UDP (direct in cluster-flor). Cluster-flor's connection manager runs both adapters simultaneously and picks per peer; the multi-hop forwarder works uniformly across mixed link types.
- **One or two flor vertices per node** instead of C0's one. Same `flor` binary, no `--role` flag — what makes a vertex link-shaped or cluster-shaped is its compiled config (transport_endpoint + connection_manager + which workloads it carries). A node with only UDP-link peerings runs just cluster-flor; nodes with any QUIC peerings additionally run link-flor.
- **Cluster-vertex as a new principal kind** — one per node per Florete cluster, distinct from C0's node principal. SPIFFE ID `spiffe://<cluster>/vertices/<node>/<name>` (e.g. `vertices/alpha/cluster` by convention). Used for hop-by-hop link-layer mTLS.
- **`flor-agent` as a distinct first-class workload**, not a vertex. The node has its own cluster-member identity (`nodes/<node>`); the agent is the workload that acts on the node's behalf — fetching configs, supervising vertices, reaching config-server. Like every other principal, the node delegates its mTLS to a vertex (cluster-flor).
- **`links.yaml`** — declares allowed UDP peerings between cluster-vertices (a topology bound).
- **`paths.yaml`** — declares allowed multi-hop routes at the cluster layer (a path bound). Hop sequences are operator-authored; labels and forwarding tables are produced as ctrl-plane decisions, not mgmt.
- **Three mgmt artifacts per node** — one `agent.json` plus `vertices/<name>.json` per vertex under `<node>/mgmt/`. Standard topology produces `vertices/public.json` (link-vertex) and `vertices/cluster.json` (mesh-vertex). Forwarding state (labels, forwarding table, label-bindings) is **not** here — it's in the ctrl artifact at `<node>/ctrl/vertices/cluster.json` (see [Ctrl Plane](../ctrl-plane/)).
- **`authorized_control_planes`** in mgmt — the cluster's mgmt artifact carries one or more authorized CP signing pubkeys; agents accept ctrl artifacts only if signed by one of these. In C1 there's a single CP key, operator-held.
- **Per-vertex FlorIO sockets** — each vertex that exposes FlorIO gets a socket at `/run/flor/<cluster>/vertices/<vertex-name>.sock`. The compiler derives the path from the cluster name and vertex name declared in `nodes.yaml`; no separate `cluster.yaml` config is needed.
- **Additional validator rules** — link validity, address feasibility, multi-hop path validity, principal-role coherence for services that appear as path initiators. (Label feasibility moves to ctrl-plane.)

Everything else (CA, per-node bundles, revocation, `flor agent sync --commit-timeout`, config-server-fronted distribution, naming, version-stamped envelope, SOCKS5 proxy, emergency-access safety net) is identical to C0.

## Source layout additions

```
my-cluster/
├── …                            # everything from C0 mgmt-plane
├── links.yaml                   # NEW: declared allowed peerings (topology bound)
├── paths.yaml                   # NEW: declared allowed paths (path bound — hop sequences only, no labels)
├── .gitignore                   # contains `.flor/compiled/*/ctrl/`
└── .flor/
    └── compiled/                # committed (operator runs `florctl compile` then commits)
        ├── alpha/
        │   ├── mgmt/            # signed by operator; committed
        │   │   ├── agent.json
        │   │   └── vertices/
        │   │       ├── public.json   # link-vertex named "public"
        │   │       └── cluster.json  # mesh-vertex named "cluster"
        │   └── ctrl/            # signed by CP key; gitignored
        │       └── vertices/
        │           └── cluster.json
        └── …
```

YAMLs in the repo are mgmt-source — they author bounds. The mgmt subtree is the compiled, signed projection of those bounds. The ctrl subtree alongside it is produced by the same `florctl compile` step in C1 (the ctrl producer becomes a CP service in B1+); it's gitignored because its lifecycle is "runtime state, not authoring artifact." See [Ctrl Plane](../ctrl-plane/) for what's inside `ctrl/`.

### cluster.yaml (additions only)

No new fields in C1 — the C0 shape is sufficient.

```yaml
cluster:
  name: rete-lovers                 # same as C0
  crypto:
    ca_cert: ca.crt
    signature_algorithm: ed25519
    cert_validity_days: 90
```

### nodes.yaml (additions over C0)

C1 extends each node's vertex graph with a **cluster-vertex** (`kind: mesh`, named `cluster` by convention) on top of the link layer, and broadens the link-vertex `type` enum from C0's QUIC-only to include `udp` (and, after C1, `wg`, `vless`, …). The cluster-vertex's `uses:` field names which link-vertices it relies upon for transit.

```yaml
nodes:
  alpha:
    vertices:
      - { name: public,   kind: link, type: quic, address: 1.2.3.4:4433 }
      - { name: internal, kind: link, type: udp,  address: 10.0.0.5:4433 }
      - { name: cluster,  kind: mesh, type: quic, uses: [public, internal] }
    agent_via: cluster              # which vertex hosts the agent's nodes/alpha principal
  beta:
    vertices:
      - { name: public,   kind: link, type: quic, address: 5.6.7.8:4433 }
      - { name: internal, kind: link, type: udp,  address: 10.0.0.6:4433 }
      - { name: cluster,  kind: mesh, type: quic, uses: [public, internal] }
    agent_via: cluster
  gamma:
    vertices:
      - { name: internal, kind: link, type: udp, address: 10.0.0.7:4433 }
      - { name: cluster,  kind: mesh, type: quic, uses: [internal] }
    agent_via: cluster
    internal_only: true             # not reachable from Internet; only via peers
```

The `kind: mesh` value is *structural* — it identifies the vertex code that operates over Florete-internal media (other vertices) and does hop-by-hop label-forwarding. The same kind is used for the cluster-vertex in C1 and for the future interrete-vertex; what distinguishes them is the **vertex `name`** the operator assigns and the configuration that follows from it. "Cluster-vertex" names a *convention*, not a YAML kind — the cluster-vertex is "the `kind: mesh` vertex named `cluster` on this node."

C1 conventions and constraints:

- **Exactly one mesh-vertex per node**, named `cluster` by convention. Its SPIFFE ID is `spiffe://<cluster>/vertices/<node>/<vertex-name>` — e.g. alpha's cluster mesh-vertex is `spiffe://rete-lovers/vertices/alpha/cluster`. The name is operator-chosen; `cluster` is the community convention, not a reserved or validated label. A future interrete-vertex on the same node would be `vertices/alpha/interrete` by convention.
- **`agent_via:`** on the node entry names which vertex hosts the agent's `nodes/<node>` principal. Required when the node has multiple vertices (always in C1). The agent connects to config-server through the named vertex's SOCKS5 listener.
- **Cluster-vertex `type`** is the upward interface contract. `type: quic` is what mesh vertices in C1 use; this is what makes recursion work — an upper-layer mesh-vertex consuming a lower mesh-vertex sees the same QUIC interface a wire-terminating QUIC link-vertex would expose.
- **Cluster-vertex `uses:`** lists the link-vertex names this cluster-vertex relies upon. Multiple link-vertices supports the canonical "border" case (one Internet-facing QUIC link + one private-net UDP link) and the parallel-underlay case (e.g., dual-ethernet UDP between two nodes — two `type: udp` link-vertices on each side, both in `uses:`).
- **Workloads (users, services) attach via `via:`** — each service and user device entry names its host vertex with a `via:` field. C1 convention is `via: cluster` for all workloads (migrating up from C0's single link-vertex). Link-vertices in C1 carry only cluster-vertex ↔ cluster-vertex transit.
- **No-process link-vertices are an implementation optimization, not a separate config category.** The runtime may collapse a `type: udp` link-vertex into cluster-flor's process (a UDP socket inline), or implement a `type: wg` link-vertex as an in-kernel WireGuard interface with no userspace process at all. The YAML lists every link-vertex uniformly; the agent decides what to spawn.

A node joining multiple Florete clusters appears in each cluster's repo under whatever name that cluster uses — cross-cluster multi-tenancy is handled at the repo level, not in `nodes.yaml`. The `address` ports on link-vertices must not overlap with another cluster's entries on the same external IP; the node operator is responsible for this when enrolling into multiple clusters (see [Multi-cluster on a single node](#multi-cluster-on-a-single-node) below).

### services.yaml (additions over C0)

C1 adds a required `via:` field to every service entry. Because C1 nodes have multiple vertices (link + mesh), the compiler must know which vertex hosts each service. Workloads in C1 belong on the cluster mesh-vertex by convention:

```yaml
services:
  api:
    at: alpha
    via: cluster                    # REQUIRED in C1: name of vertex on alpha that hosts api
    addr: 127.0.0.1:8000
    socks5_proxy: 127.0.0.1:18000
    groups: [api]
    roles: [api-backend]
  mongodb:
    at: beta
    via: cluster
    addr: 127.0.0.1:27017
    groups: [db]
  ssh:
    at: alpha
    via: cluster
    scope: node
    addr: 0.0.0.0:22
    groups: [admin]
```

`via:` names a vertex declared on the `at:` node in `nodes.yaml`. Mixed-topology opt-in (publishing a service in link-flor for private-network access) is `via: <link-vertex-name>` — no schema change.

### users.yaml (additions over C0)

C1 adds `via:` to each user device entry:

```yaml
users:
  alice:
    roles: [developer]
    nodes:
      - { at: alice-laptop, via: cluster }
  bob:
    roles: [devops]
    nodes:
      - { at: bob-workstation, via: cluster }
```

Each `nodes:` entry is an `{at, via}` object in C1. `at:` names the host node; `via:` names the vertex on that node that holds alice's identity material and SOCKS5 listener. In C0 `via:` was omittable (one vertex per node); in C1 it is required.

### links.yaml (new) — topology bound

```yaml
# Bound: the set of cluster-vertex peerings the CP is allowed to bring up.
# Each entry under `allowed_links` is a typed rule. Agents verify ctrl's
# active peers against the union of all rules' results.
# In C1 the only rule type is `enum`. New rule types can be added as
# additional entries in this list without reshaping the schema.
allowed_links:
  - type: enum
    links:
      - { a: alpha, b: beta,  via: public   }
      - { a: alpha, b: gamma, via: internal }
      - { a: beta,  b: gamma, via: internal }
  # Asymmetric form when the link-vertex names differ between endpoints:
  # - { a: { node: delta, vertex: ext }, b: { node: epsilon, vertex: eth0 } }
  # Future rule types — illustrative, not implemented in C1:
  # - { type: any-pair-via, nodes: [alpha, beta, gamma], via: internal }
```

`via` names a link-vertex declared on each endpoint node in `nodes.yaml`. Symmetric shorthand (`via: <name>`) requires both sides to have a link-vertex with that name; the asymmetric form `{a: {node, vertex}, b: {node, vertex}}` covers cases where the names diverge.

The validator checks that both sides' link-vertices have the same `type` (a `quic` link-vertex can only peer with another `quic` link-vertex) and that the two addresses can plausibly reach each other (public↔public, or shared internal network). The link's transport is fully determined by the link-vertices it references — there is no separate `link_type` field on the rule.

In C1 these declared peerings are the **bound** — the operator signs that *only* these peerings are allowed. The ctrl artifact's per-node `active_peers` is a subset (in C1, the full set — degenerate case). In B1+ a CP service might activate a subset based on health, with the bound still capping what's possible.

### paths.yaml (new) — path bound

```yaml
# Bound: the set of multi-hop routes the CP is allowed to provision.
# Each path entry has four fields:
#   principal  — user or service name (the initiator)
#   from       — the principal's home node; the starting mesh-vertex is inferred
#                from the principal's via: binding in services.yaml / users.yaml
#   hops       — ordered list of transit mesh-vertices. Each hop:
#                  node:    (required) the node hosting the target mesh-vertex
#                  vertex:  (optional) which mesh-vertex on that node; defaults
#                           when exactly one mesh-vertex on the node is reachable
#                           from the previous mesh-vertex via a declared link
#                  via:     (optional) source-side link-vertex name; defaults when
#                           exactly one link connects the previous mesh-vertex to
#                           this one; required only when multiple links connect
#                           the same mesh-vertex pair (e.g. two ethernet links)
#   to         — target service name; its host mesh-vertex (from its via: binding)
#                must equal the last hop's mesh-vertex
#
# In C1 every node has exactly one mesh-vertex, so vertex: always derives.
# via: derives unless two or more links connect the same adjacent mesh-vertex pair.
allowed_paths:
  - type: enum
    paths:
      # users
      - principal: alice
        from: alice-laptop
        hops: [{ node: alpha }]
        to: api
      - principal: alice
        from: alice-laptop
        hops: [{ node: beta }, { node: gamma }]
        to: kafka
      - principal: bob
        from: bob-workstation
        hops: [{ node: alpha }]
        to: ssh
      # services
      - principal: api
        from: alpha
        hops: [{ node: beta }]
        to: mongodb
      - principal: api
        from: alpha
        hops: [{ node: beta }, { node: gamma }]
        to: kafka
  # Future rule types — illustrative, not implemented in C1:
  # - { type: any-path-respecting, must_avoid: [delta], max_hops: 4 }
```

Paths declared here are the operator-signed bound on which routes are allowed. The ctrl artifact selects which paths are *currently provisioned* and assigns labels for them; agents only ever see local slices of those decisions (their own forwarding rows + label bindings) and never the cluster-wide path enumeration — see [Ctrl Plane](../ctrl-plane/) for why.

### Why typed-rule lists in the YAMLs (and where they don't apply)

The `allowed_links`/`allowed_paths` shells exist in the **bounds** YAMLs because those bounds are exactly the surface CP is allowed to pick within. Adding a new rule type later (say, "any peering between nodes in this VPC") is then a strictly additive change — agents implement the bound-checker as dispatch on the rule's `type`, and unknown rule types fail-closed.

Direct ACLs (`ingress`, `egress`) deliberately *don't* use this shell. They're pure operator authority — CP doesn't narrow ingress/egress allow lists, so there's no bound to express. Richer authoring grammars (roles, groups, future attribute-based selectors) resolve at compile time to flat allow lists in the JSON artifact. Same pattern as C0: rich YAML, flat compiled output.

## CLI additions

```
# operator (florctl) — authoring
florctl compile --node <name> [--repo <dir>] [--out <dir>]
                                            # emits <out>/<node>/{agent.json, vertices/*.json}

# node (flor) — runtime
flor vertex run --config <vertex.json>      # NEW: vertex daemon
                                            # the agent supervises one or more of these per node
```

`flor agent run --config <agent.json>` keeps its C0 behaviour and CLI shape; it just supervises N vertex instances rather than one. Exactly *how* the agent launches vertex processes — direct fork, systemd unit, future local workload-runtime interface — is intentionally not pinned in C1; the runtime API stays simple (a vertex starts when given a config, and the agent makes that happen by whatever means the platform offers).

`florctl ca sign` adds `vertices` to its `--kind` enumeration (already foreshadowed in C0). The vertex's SPIFFE name is derived from `<node>/<vertex-name>`.

## Additional validator rules

On top of C0's:

- **Vertex graph validity** — every node declares at least one mesh-vertex (`kind: mesh`, named `cluster` by convention in C1) and at least one link-vertex; every `uses:` reference resolves to a vertex name declared on the same node; vertex names are unique within a node. C1 graphs are trees (one mesh root above link-vertex leaves); the schema does not require acyclicity in general — see [Reasoning > Why the vertex graph is a directed graph, not a DAG](#reasons-behind-the-c1-additions).
- **Workload vertex binding** — every service's `via:` and every user device entry's `via:` must name a vertex declared on the respective `at:` node. The `agent_via:` field on a node entry must name a vertex declared on that node. In C1 these fields are required (multiple vertices per node); omitting them is an error ("specify `via:` to disambiguate among `<list of vertex names>`"). Validator also warns (not errors) when `via:` references a `kind: link` vertex — that is a valid mixed-topology opt-in but unusual enough to surface.
- **Link validity** — every link is between two distinct, known nodes; the `via` reference resolves to a link-vertex on each side (symmetric shorthand or asymmetric form); both link-vertices have the same `type`.
- **Address feasibility** — link pairings use addresses that can actually reach each other (public↔public, or shared internal network). Internal-only nodes can only pair with peers on the same internal network.
- **Link-type coherence** — `type: udp` link-vertices may only peer over shared internal networks (not over Internet-facing addresses), since they carry no link-layer crypto. `type: quic` link-vertices are allowed everywhere.
- **Multi-hop path validity** —
  - `principal` is a known user or service name. `from` is a known node; for users it must appear in `users.yaml::nodes`; the starting mesh-vertex is the one declared in the principal's `via:` binding.
  - Each `hops[].node` is a known node. `hops[].vertex` (when present) names a mesh-vertex declared on that node; when absent, the compiler resolves it: it must be uniquely determined by the links in `allowed_links` that connect the previous mesh-vertex to any mesh-vertex on the hop node. If the lookup is ambiguous, validation fails with "specify `vertex:` to disambiguate."
  - `hops[].via` (when present) names a link-vertex on the **previous** node (the source side of this hop). When absent, there must be exactly one link in `allowed_links` connecting the previous mesh-vertex to this hop's mesh-vertex; if multiple links connect the same pair, validation fails with "specify `via:` to disambiguate."
  - Every adjacent mesh-vertex pair in the resolved path (origin → hop₁ → … → last hop) has a declared link in `allowed_links`.
  - `to` is a published service. Its host mesh-vertex (from its `via:` binding in services.yaml) must equal the last hop's resolved mesh-vertex; for on-origin services, equal the origin mesh-vertex.
- **Principal role coherence** — services that appear as path initiators must have a `role` declared.

(Label feasibility is now a ctrl-plane concern — the label allocator runs against the operator-declared paths and surfaces any infeasibility there. See [Ctrl Plane](../ctrl-plane/).)

## Compile step additions

`florctl compile` produces **two artifact subtrees per node** in C1: a signed mgmt subtree (`<node>/mgmt/`) and a CP-signed ctrl subtree (`<node>/ctrl/`). The mgmt subtree contains one `agent.json` plus one `vertices/<name>.json` per vertex (standard topology: `vertices/public.json` for the link-vertex and `vertices/cluster.json` for the mesh-vertex), all sharing the common envelope inherited from C0 (including `plane: "mgmt"` and the operator signature — see [C0's envelope spec](../../c0-tended-tunnels/mgmt-plane/validate-and-compile#compile-step)). The ctrl subtree contains only what's CP-shaped (forwarding tables, label bindings) and is documented separately in [Ctrl Plane](../ctrl-plane/).

This page covers the mgmt envelope and the per-vertex mgmt payloads. Mgmt envelope:

```json
{
  "schema_version": "1.0",
  "plane": "mgmt",
  "kind": "agent",          // or "vertex"
  "version": 42,
  "node": "alpha",
  "generated_at": "2026-04-20T12:00:00Z",
  "payload": { ... },
  "signature": { "alg": "ed25519", "key_id": "operator:<fingerprint>", "value": "<base64>" }
}
```

`plane` and `kind` are orthogonal: `plane` is the decision-authority axis (`"mgmt"` signed by the operator today; `"ctrl"` reserved for the unsigned control-plane stream introduced in B1+), `kind` distinguishes the supervisor artifact from the data-plane vertex artifacts. C1 stays at `plane: "mgmt"` for everything; the CP stream lands later.

### Workloads & IO

Both vertex configs use a unified `workloads` shape — one entry per local principal/target. Each workload has zero or more `io` channels, each with a direction from flor's PoV (matching C0 scope's "Inbound/Outbound" usage):

- `inbound` — the workload connects *into* flor here; flor originates outgoing connections on its behalf. Workload acts as a **principal**.
- `outbound` — flor connects *out* to the workload here; flor delivers incoming connections. Workload acts as a **target** (service).
- `bidirectional` — single channel does both. Used for FlorIO between recursive flor layers, where one socket carries Connect requests downward and inbound deliveries upward.

A workload entry:

```json
{
  "spiffe_id": "spiffe://rete-lovers/services/api",
  "identity": { "cert_path": "...", "priv_path": "..." },
  "io": [
    { "kind": "tcp",    "upstream": "127.0.0.1:8000",  "direction": "outbound" },
    { "kind": "socks5", "listen":   "127.0.0.1:18000", "direction": "inbound"  }
  ]
}
```

`io.kind` is open-ended; today's inventory is `socks5`, `tcp`, `florio` (the latter is new in C1). New northbound interfaces add new kinds without restructuring the schema.

### `vertices/public.json` payload

The link-vertex config file is named after the vertex's `name:` field in `nodes.yaml` — `public` in the standard topology. The compiler derives the filename from the vertex name; operators never pick it manually.

```json
{
  "ca_cert_path": "ca.crt",
  "transport_endpoint": { "type": "quic" },
  "connection_manager": { "adapter": "udp", "listen": "0.0.0.0:4433" },
  "workloads": [
    {
      "spiffe_id": "spiffe://rete-lovers/vertices/alpha/cluster",
      "identity":  { "cert_path": "cv-alpha.crt", "priv_path": "cv-alpha.key" },
      "io": [
        { "kind": "florio", "socket": "/run/flor/rete-lovers/vertices/public.sock", "direction": "bidirectional" }
      ]
    }
  ],
  "ingress": [
    {
      "target_spiffe_id": "spiffe://rete-lovers/vertices/alpha/cluster",
      "allow": [
        "spiffe://rete-lovers/vertices/beta/cluster",
        "spiffe://rete-lovers/vertices/gamma/cluster"
      ]
    }
  ],
  "egress": [
    {
      "target_spiffe_id": "spiffe://rete-lovers/vertices/beta/cluster",
      "via": { "udp_peer": "5.6.7.8:4433" },
      "allow": ["spiffe://rete-lovers/vertices/alpha/cluster"]
    },
    {
      "target_spiffe_id": "spiffe://rete-lovers/vertices/gamma/cluster",
      "via": { "udp_peer": "10.0.0.7:4433" },
      "allow": ["spiffe://rete-lovers/vertices/alpha/cluster"]
    }
  ]
}
```

Things worth pointing out about this shape:

- **The cert+key are link-flor's loaded material, not its identity.** `vertices/alpha/cluster` is a workload entry that link-flor serves and acts on behalf of. From link-flor's perspective it's just a service name with a bidirectional FlorIO channel — link-flor has no idea what a cluster-vertex semantically is. The structural mirror to C0 flor holding alice's cert+key is exact.
- **No top-level `florio` block.** FlorIO sockets are properties of individual workloads' `io` entries, derived from the vertex name. The socket path `/run/flor/<cluster>/vertices/<vertex-name>.sock` is a compiler convention, not a config field.
- **`ingress`/`egress` work just like C0's.** The link layer has the same access-control shape as the cluster layer; flor doesn't distinguish which mTLS layer it's enforcing.
- **No forwarding table, no labels, no paths.** Multi-hop forwarding lives entirely in cluster-flor.

Mixed-topology case (publishing a regular service in link-flor for private-network use): the service's workload entry simply moves from `cluster.json` into the link-vertex's config file (`public.json` in the standard topology), with no schema change. The compiler decides placement based on the service's `via:` field.

### `cluster.json` payload

```json
{
  "ca_cert_path": "ca.crt",
  "transport_endpoint": { "type": "quic" },
  "connection_manager": {
    "adapters": [
      { "name": "via-link", "adapter": "florio", "socket": "/run/flor/rete-lovers/vertices/public.sock" },
      { "name": "internal", "adapter": "udp",    "listen": "10.0.0.5:5544"     }
    ]
  },
  "allowed_peers": [
    {
      "type": "enum",
      "peers": [
        {
          "spiffe_id": "spiffe://rete-lovers/vertices/beta/cluster",
          "via": { "adapter": "via-link" }
        },
        {
          "spiffe_id": "spiffe://rete-lovers/vertices/gamma/cluster",
          "via": { "adapter": "internal", "udp_peer": "10.0.0.7:5544" }
        }
      ]
    }
  ],
  "workloads": [
    {
      "spiffe_id": "spiffe://rete-lovers/nodes/alpha",
      "identity":  { "cert_path": "alpha.crt", "priv_path": "alpha.key" },
      "io": [
        { "kind": "socks5", "listen": "127.0.0.1:1080", "direction": "inbound" }
      ]
    },
    {
      "spiffe_id": "spiffe://rete-lovers/services/api",
      "identity":  { "cert_path": "api.crt", "priv_path": "api.key" },
      "io": [
        { "kind": "tcp",    "upstream": "127.0.0.1:8000",  "direction": "outbound" },
        { "kind": "socks5", "listen":   "127.0.0.1:18000", "direction": "inbound"  }
      ]
    }
  ],
  "ingress": [
    {
      "target_spiffe_id": "spiffe://rete-lovers/services/api",
      "allow": ["spiffe://rete-lovers/users/alice"]
    }
  ],
  "egress": [
    {
      "target_spiffe_id": "spiffe://rete-lovers/services/mongodb",
      "via": { "first_hop": "spiffe://rete-lovers/vertices/beta/cluster" },
      "allow": ["spiffe://rete-lovers/services/api"]
    }
  ],
  "authorized_control_planes": [
    {
      "spiffe_id": "spiffe://rete-lovers/control-planes/primary",
      "pub_key": "<base64-ed25519>"
    }
  ]
}
```

The agent's principal — `nodes/alpha` — appears as a workload here so the agent can dial config-server through cluster-flor like any other principal. This is the mechanism that makes the agent's bootstrap chain work: agent → cluster-flor's SOCKS5 → mTLS to config-server. It's also why the agent doesn't need to hold any cert+key itself.

`connection_manager.adapters[]` is a **list**, one entry per active link type, each with a stable `name` that per-peer `via.adapter` fields reference. The example above shows alpha running both adapters: `via-link` (FlorIO down to link-flor for QUIC peerings) and `internal` (a bare UDP socket for the trusted-LAN peering with gamma). A node with only QUIC peerings would have just the `florio` adapter; a node with only UDP peerings would have just the `udp` adapter and **no link-flor at all**. Other adapter values (`wg` and similar for third-party links) are foreshadowed in [Scope > Link Protocols](../scope#link-protocols) and [Data Plane](../data-plane/), not implemented in C1.

`allowed_peers` is a **typed-rule bound list**, mirroring `allowed_links`/`allowed_paths` in the YAMLs. Each rule produces a set of allowed peers (SPIFFE ID + adapter selection); the agent's effective allowed-peer set is the union across rules. CP-shaped decisions (which subset is currently *active*, which forwarding rows reference which peer) are verified against this bound at the agent — see [Ctrl Plane](../ctrl-plane/).

In C1 the only rule type is `enum`; the rule body lists peers explicitly. The `via` per peer (adapter + dial info) is fixed by the operator — CP doesn't switch a peer between QUIC and bare UDP at runtime. CP can only narrow *which* allowed peers it actually uses.

`ingress.allow` and `egress.allow` are **direct flat allow lists** of authorized SPIFFE IDs. They are not bounds and not typed rules — they're pure operator authority. Compile time resolves richer YAML grammar (roles, groups, future selectors) into these flat lists; the agent's check is a simple `principal ∈ allow`. CP cannot narrow ingress/egress, so there's no bounds shell to provide.

Per-allow forwarding labels (`local_label`) and the `forwarding_table` itself are **not in this artifact** — they're CP-shaped state and live in the ctrl artifact (see [Ctrl Plane](../ctrl-plane/)). The mgmt artifact carries only what the operator authored: who may reach what, which peers are allowed, and (for egress) what the first hop is.

`authorized_control_planes` lists CP signing identities the operator has approved; agents accept ctrl artifacts only if signed by one of these pubkeys. In C1 there's a single entry — the operator's CP key, held alongside (but separately from) the operator's mgmt signing key. In B1+ this list grows to one or more CP services.

### `agent.json` payload

```json
{
  "ca_cert_path": "ca.crt",
  "vertices": [
    { "name": "public",  "config": "vertices/public.json"  },
    { "name": "cluster", "config": "vertices/cluster.json" }
  ],
  "control_plane": {
    "principal":     "spiffe://rete-lovers/nodes/alpha",
    "config_server": "spiffe://rete-lovers/services/config-server"
  }
}
```

The agent supervises every vertex listed under `vertices`. To reach `config-server`, the agent acts as the principal `nodes/alpha`; it finds the SOCKS5 port for that principal by scanning the referenced vertex configs for a matching workload's `inbound` channel. Paths in the artifact (`ca_cert_path`, `cert_path`, `priv_path`) are scope-relative and resolved by the agent against the cluster install root (`~/.flor/clusters/<scope>/`) at startup — see [C0's compile step](../../c0-tended-tunnels/mgmt-plane/validate-and-compile#compile-step) for the convention.

The number and shape of vertices is fully driven by `vertices`. The standard mixed-topology has two (link + cluster); a node with only `udp`-type links runs cluster-flor alone (one vertex, no link-flor); an enrollment-bootstrap config (single-vertex, direct UDP to mgmt01, just enough for the first sync) is also one. The agent doesn't care — it enumerates whatever it's told to.

Label allocation, forwarding tables, and the per-allow label bindings are ctrl-plane state, not mgmt — see [Ctrl Plane](../ctrl-plane/) for that machinery.

## Identity model additions

C1 adds two new principal kinds on top of C0's users, services, and nodes:

| Kind | In C0 | In C1 | SPIFFE ID | Used for |
|---|---|---|---|---|
| Cluster vertex | No | **Yes** | `spiffe://<cluster>/vertices/<node>/cluster` | Hop-by-hop link-layer mTLS (one per node per cluster, named `cluster` by convention) |
| Control plane | Reserved | **Yes** | `spiffe://<cluster>/control-planes/<name>` | Signing authority for ctrl artifacts (one or more per cluster; in C1 a single operator-held key) |

A useful "owned-by / held-by" view across all four principal kinds:

| Kind | Principal (identity owner) | Cert+key held by (the vertex on its behalf) |
|---|---|---|
| node | the node (the host as cluster member) | cluster-flor (via the `nodes/<node>` workload entry there) |
| cluster-vertex | cluster-flor (qua principal in the link layer) | link-flor |
| user | the human (their workload) | cluster-flor |
| service | the service workload | cluster-flor (or link-flor in mixed-topology setups) |

A vertex always *holds* identity material on someone else's behalf — that's the recursive pattern from the [HLD](../../overview/high-level-design#vertex). Cluster-flor does the cluster-layer mTLS for its workloads; link-flor does the link-layer mTLS for cluster-flor (its sole standard-topology workload). No vertex is a principal at its own layer; every vertex is a principal at the layer below.

The `nodes/<name>` principal is the node's identity *within a specific cluster's trust domain* — the cluster's logical view of this computing host. **flor-agent is not the principal; it is the workload that acts on the node's behalf**, just as alice's process acts on alice's behalf. cluster-flor holds the cert+key and runs mTLS for `nodes/<name>`, exactly as it does for user and service principals. The agent's job is to wield that identity: fetch configs, supervise vertices, and reach config-server through cluster-flor's SOCKS5 listener.

Node-identity vs. cluster-vertex-identity aren't merged because they serve different layers and are held by different vertices — `nodes/<name>` is for control-plane traffic (agent ↔ config-server), `vertices/<node>/cluster` is for hop-by-hop mesh transit mTLS. A node joining multiple Florete clusters has one `nodes/<name>` principal per cluster (in each cluster's own trust domain) and one flor-agent process per cluster acting on its behalf.

Enrollment flow is unchanged: `florctl issue-bundle --node alpha --cluster <config-server-url>` packages alpha's node identity *and* alpha's cluster-vertex identity in the same per-node bundle, alongside every user/service principal hosted at alpha. The compiler knows to add a `vertices/<node>/cluster` principal for every node in C1 (derived from the cluster mesh-vertex's name and its host node); operators don't declare it anywhere. SPIFFE IDs are derived from the vertex name, the host node, and the cluster trust domain.

CP signing identities are issued separately on the operator's own machine (`florctl ca sign --kind control-planes --name primary` produces the cert+keypair that signs ctrl artifacts). The CP key is *not* shipped in per-node bundles — only its public counterpart, embedded in `authorized_control_planes` of the mgmt artifact. This keeps the CP private key purely on the operator's workstation in C0/C1 (and on whichever host runs the CP service in B1+); no node ever sees it.

The bundle may also include a current `agent.json` + per-vertex configs from the latest published state — sufficient for the agent's first run with no prior sync. Subsequent `flor agent sync` calls pull whatever the cluster currently runs (which may be richer or simpler than what was bundled).

## Multi-cluster on a single node

The foundation — cluster scope, per-scope install root, one agent per cluster, scope-relative artifact paths, local listen-address ownership — is established in [C0's multi-cluster section](../../c0-tended-tunnels/mgmt-plane/#multi-cluster-on-a-single-node). C1 introduces two additional concerns specific to the mesh layer.

**FlorIO socket path.** The socket between cluster-flor and a link-flor vertex is at `/run/flor/<cluster-name>/vertices/<link-vertex-name>.sock`. Both segments are scope-distinct by construction: `<cluster-name>` separates clusters, and `<vertex-name>` separates vertices within a cluster. Two clusters' stacks on the same host use different socket paths as long as their cluster names differ — which they will unless two independent operators happen to share a name, in which case the node operator overrides the local scope name via CLI (`flor enroll <bundle> --as <scope>`).

**UDP connection-manager listeners and external addresses.** Cluster-flor's bare-UDP adapter (`connection_manager.adapters[].listen`) and link-flor's QUIC-link UDP listener only appear on server nodes — nodes with a declared external `IP:port` in `nodes.yaml`. Server nodes are by definition admin-managed. Two clusters binding the same external `IP:port` on the same host must be coordinated by the IT admin responsible for that host; this is the same constraint as any two services sharing a server. User devices have no fixed UDP listen address and are unaffected. B1+ direction: node-side port-availability advertisement, so the cluster's mgmt-plane knows which external ports the node has available rather than relying on operator coordination (see [C0 multi-cluster section](../../c0-tended-tunnels/mgmt-plane/#multi-cluster-on-a-single-node)).

## Explicitly deferred (in addition to C0's list)

- `florctl compile --derive-paths` — declarative path derivation from topology + access. Ships post-C1 once the topology model is stable.
- Priority / fallback paths.
- 3rd-party connection-manager adapters (WireGuard, VLESS, …) and the identity-projection rules they need. Architectural placeholder in [Scope > Link Protocols](../scope#link-protocols) and [Data Plane](../data-plane/); no implementation in C1, but the multi-adapter `connection_manager.adapters[]` shape is exercised by the QUIC + UDP pair so adding a third adapter type is purely additive.
- Mixed-vertex publication (services in link-flor instead of cluster-flor). Now a corner case rather than a planned optimisation, since UDP links cover the private-network performance story without splitting publication. Schema still allows it; YAML surface and validator rules deferred indefinitely.

## Reasoning

Captured justifications for the C1-specific additions. See the [C0 reasoning](../../c0-tended-tunnels/mgmt-plane/reasoning) for everything else.

<Accordions type="single">
  <Accordion title="Reasons behind the C1 additions">

### Why "kind: mesh" rather than "kind: cluster"

Two axes are worth keeping separate when classifying a vertex:

- **Kind** (structural — what the code does): does it operate over external medium and terminate a transport on the wire (link), or does it operate over Florete-internal medium and do hop-by-hop label-forwarding (mesh)? This is a property of the vertex implementation.
- **Role** (configuration — where it sits in the architecture): cluster-vertex, interrete-vertex, … . This is an assignment made at deploy time when the operator wires `uses:` and the per-cluster scope.

C0's design notes used "cluster-vertex" as if it were a kind, but it's really a role. The interrete-vertex (a future upper layer between clusters) shares the structural behavior with the cluster-vertex completely — same code, same forwarding, same recursion. They differ only in which scope they participate in and what their peers are. Naming the kind `mesh` (after the defining behavior — multi-hop label-forwarded mesh transit) keeps the structural axis crisp and lets new roles plug in as configuration rather than as new kinds.

This is also what makes the recursion property in `type` work cleanly: a `kind: mesh, type: quic` vertex exposes a QUIC interface upward, indistinguishable to consumers from a `kind: link, type: quic` wire-terminating vertex. Recursion is the consequence; mesh is the structure.

### Why the vertex graph is a directed graph, not a DAG

Today's C1 graphs are trees (mesh root, link leaves) and the obvious mental model is "vertices form a DAG — `uses:` is acyclic." Worth resisting that simplification at the schema level.

A concrete future case that breaks acyclicity: a **uni-directional link-vertex**. Imagine a satellite-downlink-style link that physically only carries packets one way. The cluster-vertex above uses this link-vertex for the forward direction. The link-vertex itself, having no return channel, needs the cluster layer to obtain its backward direction — so the link-vertex `uses:` the cluster-vertex too. Result: a 2-cycle (link ↔ mesh), each using the other for a different purpose. Cluster forwards data through link going out; link hands return signaling back to cluster going in.

Other shapes are imaginable (radio-relay rings, asymmetric tunnels with crypto on one side only), and the unifying property is "directed graph, cycles allowed." Pinning to DAG at the schema level would close off these designs unnecessarily.

C1 does not implement cycles — the validator can require acyclicity *in C1* as a temporary simplification. The schema and the agent's traversal logic should not assume it.

### Why the agent is separated from the vertex

C0's single flor process did three jobs at once: it ran the data-plane vertex, it pulled state from config-server, and it handled node-level control-plane traffic. Conflating these works for one process per node but breaks the moment there's more than one — *which* vertex talks to config-server? With a recursive stack the answer ("just one of them, by config") becomes a design question; cleaner to give the node a first-class cluster-member identity (`nodes/<node>`) with the agent as the dedicated workload that acts on its behalf, and to treat every vertex as a pure data-plane component.

The agent doesn't *hold* the node cert+key — cluster-flor does, exactly as it holds user and service identities. The node is the principal; the agent is the workload that wields that identity (fetching configs, reaching config-server via cluster-flor's SOCKS5 listener), exactly as alice's process acts on alice's behalf. The principal/vertex distinction is the same mechanism, applied to a workload that happens to live alongside its serving vertex on the same machine.

This also maps to the HLD's [Execution Plane Agent concept](../../../overview/high-level-design#execution-plane-green): the agent is the per-node supervisor of workloads, and in C1 the workloads it supervises are flor vertex processes plus (eventually) user and service workloads.

How exactly the agent launches vertex processes — direct fork, systemd unit, future local workload-runtime interface — is intentionally not pinned in C1. The runtime API stays simple: a vertex starts when given a config; the agent makes that happen by whatever means the platform offers. The shape resembles kubelet's relationship to its container runtime; the analogue here is intentional but the WRI design is deferred.

### Why cluster-flor's identity lives in link-flor's config

Cluster-flor publishes itself as a service in the link layer — it *is* the cluster-vertex principal there. By the recursive principle, that means link-flor (its serving vertex at the link layer) holds the cluster-vertex cert+key and runs the QUIC handshake with peer cluster-vertices on cluster-flor's behalf.

The shape is identical to C0 flor holding alice's cert+key: alice owns the identity, flor holds the material and acts on alice's behalf. We never confuse alice with flor; we shouldn't confuse cluster-flor with link-flor either. Both pairings are instances of the same principal/vertex relationship, just at different layers.

A consequence worth being explicit about: link-flor's config is **vocabulary-neutral**. It speaks of `workloads`, `ingress`/`egress`, SPIFFE-ID names, and FlorIO sockets — never of "cluster vertex". From link-flor's PoV, `vertices/alpha/cluster` is just a workload it serves with bidirectional FlorIO. Mixing in a regular service there (a private-network optimisation) is a one-line YAML change with no schema impact.

Publication is **static-config-only** in C1: link-flor's local services are baked into its compiled JSON, not announced dynamically over FlorIO. This is a deliberate simplification — keeps everything publishable under the signed-config model and avoids designing the Publish side of FlorIO before we need it. Connect remains the only dynamic FlorIO verb.

### Why one `workloads` list and an `io[]` direction

C0 had `local_users` (with `socks5_proxy`) and `local_services` (with `upstream_addr`, sometimes also `socks5_proxy`). The split conflated principal-vs-target with workload-kind; C1's cluster-vertex case broke it (a cluster-vertex is both, on a single FlorIO channel).

Unifying to `workloads[].io[].direction` resolves the asymmetry: a workload is a principal if it has at least one `inbound` channel, a target if it has at least one `outbound` channel, both if both. FlorIO becomes a `bidirectional` channel — one socket, both directions, no mechanism duplication. The schema also stops privileging SOCKS5 and TCP-upstream as the only northbound shapes; new `io.kind` values can be added without restructuring.

### Why three compiled artifacts per node

C1 runs an agent plus N flor vertices (standard topology: 2). Each is a separately-spawned process with its own config; bundling them into one artifact would force the agent to demultiplex and to know vertex-internal schemas, which it has no reason to. Splitting into `agent.json` + `vertices/*.json` keeps each consumer's contract minimal: the agent reads its own list and supervises; each vertex reads its own config and runs.

Operator YAML doesn't expose this fan-out. The compiler is the only thing that thinks in vertex configs; operators think in nodes, links, services, paths.

### Why two link types from C1 (and not just QUIC)

A single-link-type C1 forces a bad tradeoff. QUIC-only is right for Internet-facing peerings (NAT traversal, metadata-resistance, mTLS at the link layer) but burns CPU and adds a second crypto layer for peerings that already cross only trusted private network — datacenter racks, office LANs, on-vehicle Ethernet. UDP-only would leave the Internet-facing case unsolved.

The clean answer is to make link type a per-link choice, with both adapters present in cluster-flor's connection manager from the start. From the workload's PoV nothing changes — the mesh looks like one mesh, services are published once in cluster-flor regardless of which underlay each hop uses. From cluster-flor's PoV the multi-hop forwarder takes a peer cluster-vertex SPIFFE ID, looks it up in `allowed_peers` for the adapter selection, and dials through the right adapter; the forwarder doesn't know or care whether the hop is QUIC-via-link-flor or bare-UDP-direct.

A consequence worth being explicit about: bare UDP links **don't need a link-flor process**. A node with only UDP-type peerings runs cluster-flor alone, with a `udp` adapter directly bound to a UDP socket. This is also why publication should always go through cluster-flor in the standard topology — using link-flor publication for "private network optimisation" was the wrong solve; the right solve is a UDP link, which keeps the recursive principle intact and the mesh story uniform.

Implementing both adapters from C1 also exercises the multi-adapter shape of `connection_manager.adapters[]` in production. Adding a third adapter (WireGuard, VLESS, …) later is then purely additive — no reshape of the schema, no migration of existing configs.

Why bare UDP and not "QUIC without crypto"? Because the simplest possible thing wins: a UDP socket is a UDP socket. Anything with QUIC framing pays per-packet header tax that's pure waste in the trusted-LAN case. Reserving "QUIC-but-cheaper" as a future option is fine; in the meantime the bare-UDP adapter is trivial to implement, trivial to reason about, and gives operators the high-performance internal-mesh story they need.

### Why mesh-vertex SPIFFE is `vertices/<node>/<name>`, not `vertices/<role>/<node>`

Identity should answer "who is this principal?", not "what is it configured to do?" Other principals in our trust domain follow this rule: alice's `users/alice` SPIFFE ID doesn't carry her role (`developer`); api's `services/api` doesn't carry its role (`api-backend`). Roles, groups, intent — these are configuration that resolves to ACLs at compile time, not facets of identity.

An earlier iteration encoded role as a SPIFFE path segment (`vertices/cluster/<node>`). That broke the identity/configuration separation other principals respect. Switching to `vertices/<node>/<name>` fixes the leak: node-scoping is structural (a vertex literally lives on a node), name is the local identifier within the node, and role/intent — when needed — lives in YAML metadata (deferred until a feature actually demands it).

This also mirrors `services/<node>/<service>` exactly. Mesh-vertices are, conceptually, services — they provide IPC for the mesh to function. Treating their SPIFFE form the same way as node-scoped services keeps the trust-domain naming uniform.

Convention names (`cluster` for the cluster mesh-vertex; `interrete` for the B1+ interrete-mesh-vertex) are operator habits, not validator rules — same as `<service>` names under `services/<node>/`.

### Why workload binding to vertex is explicit

C0 has one vertex per node; the question "which vertex hosts this workload?" had a single answer and could be implicit. C1 has multiple vertices per node, and the question genuinely admits more than one answer (workloads can be hosted on the cluster mesh-vertex in the standard topology, or on a link-vertex in mixed-topology setups, or — in B1+ — on different mesh-vertices for different scopes).

Explicit `via:` in services.yaml and users.yaml forces the operator to express the choice rather than relying on a derivation rule the compiler picks unilaterally. The default-when-unique rule (one rule, applied across all milestones) keeps C0 ergonomic without weakening C1's correctness story.

This also makes the previously-deferred "mixed-vertex publication" case trivial: an operator who wants a service published in link-flor for private-network optimisation just writes `via: <link-vertex>`. No special schema, no migration, no flag — it falls out of the binding mechanism we'd have written anyway.

Future bound-derivation rules (CP picks `via:` based on health, scope, or selector grammar) plug into the same mechanism: the authoring surface stays operator-explicit; the compiler may grow derivation when YAML grammar grows.

### Why typed-rule bounds (and only for CP-narrowable surface)

The typed-rule list shape (`{type: <discriminator>, ...}` items) wraps three things in C1: `allowed_links`, `allowed_paths`, and `allowed_peers`. The common thread is that each is a **bound a CP can narrow within** — CP picks which peerings to actually bring up, which paths to provision, which peers to use this minute. The shell is forward-compat insurance: a future bound type (e.g. `any-pair-via`, `any-path-respecting-constraint`) plugs in as a new `type` value without reshaping the schema; agents that don't recognize a `type` fail-closed.

We deliberately *don't* apply the shell to `ingress.allow`/`egress.allow`. Those aren't bounds — CP doesn't narrow them. They're pure operator authority: a flat SPIFFE-ID list resolved at compile time from richer YAML grammar (roles, groups, future attribute-based selectors). Wrapping a flat allow list in a one-rule typed shell would be cargo-cult — the CP-narrowing test it's meant to enable doesn't apply.

The pattern of "discriminator-tagged bound types" is proven elsewhere — Kubernetes admission policies, OPA policy types, AWS IAM condition operators. C1's bounds are tiny (one rule type, `enum`); the policy-language work is deferred. The *shell* is what later policy growth plugs into without forcing a schema migration.

### Why paths are over mesh-vertices, not nodes — and why the notation includes link specification

The obvious mental model is that a path is a sequence of nodes. It's wrong in two ways.

**Paths are between mesh-vertices.** The thing doing the multi-hop forwarding in Florete is cluster-flor — the `kind: mesh` vertex on each node. Link-flor (the `kind: link` vertex) terminates the wire transport, but it has no forwarding logic; it hands traffic up to cluster-flor over FlorIO. A path, then, is a sequence of cluster-flor instances on successive nodes — a sequence of mesh-vertices, not nodes. In C1 this distinction is invisible because every node has exactly one mesh-vertex (named `cluster` by convention), making node and mesh-vertex 1:1. But the identity of the thing being named is still the mesh-vertex, and the notation must say so.

Why it matters in B1+: a node could host both a `cluster` mesh-vertex (for the local cluster) and an `interrete` mesh-vertex (for inter-cluster forwarding). A path that naively references nodes would be ambiguous about which mesh-vertex to traverse at that node — the two vertices serve different scopes and the path shouldn't mix them. Naming mesh-vertices directly makes the scope unambiguous and leaves no room for the compiler to guess wrong.

**The hop notation also carries link specification.** A path hop from mesh-vertex A to mesh-vertex B is realized by a specific link declared in `allowed_links`. In C1 there is usually only one link between any given pair of mesh-vertices, so the link derives automatically. But "usually" is not "always": two servers could have two separate Ethernet connections between them, each configured as a distinct `type: udp` link-vertex pair — both feeding into the same cluster-vertex pair. In that case the path must say which physical link the route should use, because the choice affects latency, capacity, and which physical adapter carries the traffic.

The `via:` field on a hop addresses exactly this: it names the source-side link-vertex for that hop. It is *not* triggered by a node having multiple link-vertices in general — having both a `public` QUIC link-vertex and an `internal` UDP link-vertex on alpha does not require `via:` on the alpha→beta hop, as long as only one of them participates in the alpha/cluster ↔ beta/cluster link. `via:` is only needed when the same mesh-vertex pair is connected by more than one declared link.

Both fields (`vertex:` for mesh-vertex disambiguation, `via:` for link disambiguation) apply the same default-when-unique rule used everywhere else in the design: omit when unambiguous, specify when not. In C1 `vertex:` is always omittable. `via:` is omittable unless the operator explicitly provisions redundant physical links between the same pair of nodes — a deliberate, unusual topology choice. Both fields are reserved in the schema so B1+ can express complicated situations without a format migration.

### Why labels and forwarding are ctrl-plane, not mgmt

Labels and forwarding tables are intrinsically forwarding-implementation, not security-relevant facts: whether a label is 17 or 42 has no access-control consequence — security comes from the `ingress.allow` check at the target. Tying labels to operator signing cadence forces re-signing on every label change, which is dumb in C1 and impossible in B1+ (where CP changes paths in real time). They belong in the unsigned-but-bounded ctrl stream, even in C1 where the producer is `florctl compile`.

Treating C1 this way also avoids a schema migration at C1→B1+. See [Ctrl Plane](../ctrl-plane/) for the actual mechanics of how labels are allocated, signed (by CP), and verified.
