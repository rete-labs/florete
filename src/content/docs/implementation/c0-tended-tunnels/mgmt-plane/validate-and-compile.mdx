## Validator Rules

`flor validate` is the only safety net against hand-edit errors, so it must be strict.

1. **Manifest shape** - every YAML document in the source bundle is a Florete resource with `apiVersion`, `kind`, `metadata.name`, and `spec`. Supported `kind` values in C0 are `FloreteCluster`, `FloreteNode`, `FloreteService`, `FloretePrincipal`, `FloreteRole`, and `FloreteGroup`.
2. **Schema** - every resource matches the JSON Schema for its `(apiVersion, kind)`.
3. **Cross-references** - every node/service/group/role name referenced exists and is unique across the whole cluster.
4. **Principal registry** - every principal name (principal resources, service resources, node resources; vertices in C1+) is unique across all principal kinds; no name collision across YAML files. Cross-kind disjointness also keeps the `.rete` DNS namespace unambiguous (see [ADR-0006](../../../../contributing/adr/0006-impl-mtls-in-quic-endpoint) for the `dns::format` mapping): without it, e.g. a service and a vertex sharing a short name on the same node would both render to `<name>.<node>.<cluster>.rete`.
5. **Enrollment log consistency** - every `FloretePrincipal`, every principal-bearing `FloreteService`, and every `FloreteNode` has a matching sign-event in `enrollment.log` that hasn't been superseded by a revoke-event.
6. **Service placement** - every service's `spec.at` field names a known `FloreteNode`.
7. **Management-node integrity** - services named `config-server` and `config-publisher` must exist in groups `config-read` and `config-write` respectively, both hosted on the same node (the management node) with a publicly-reachable link-vertex (a `vertexClass: link, transport: quic` entry with an `address`). If missing or malformed, validator fails with a clear message rather than silently producing artifacts that can't sync.
8. **Reserved-name protection** - `node` and `operator` (`FloreteRole`), `config-read` and `config-write` (`FloreteGroup`) must be present in the source bundle with their canonical `allow` / member definitions. Validator fails if they're missing or redefined with different structure. The compiler never writes source manifests - these live in the repo as part of the initial cluster template.
9. **Operator presence** - at least one `FloretePrincipal` must have `operator` in `spec.roles`. Without it, no one can call `config-publisher`, so the cluster can't receive new state after the initial bootstrap - worth catching at validate time.
10. **Mgmt-plane signer presence** - the `FloreteCluster` resource's `spec.signers.mgmt.keys` list must be non-empty and every referenced cert file must exist under `certs/management-planes/`, parse as X.509, carry a SPIFFE URI SAN matching `spiffe://<cluster>/management-plane/<key.name>`, be signed by the cluster CA, and have a corresponding sign-event in `enrollment.log`. Without at least one signer, no key exists to sign compiled artifacts; without the SAN check, a misfiled cert could pass undetected.
11. **Vertex graph & reachability feasibility** - every node declares at least one link-vertex (`vertexClass: link`); in C0, exactly one with `transport: quic`. Every node hosting a workload service (other than `config-read` services) must have at least one link-vertex with an Internet-reachable `address`. Initiator-only nodes (user laptops) may omit `address` on their link-vertex; referencing them as a service host is an error.
12. **Workload vertex binding** - every workload's host vertex must resolve. For services: `spec.at` names a known node; `spec.via` (when present) names a vertex declared on that node. For principal device entries: `spec.nodes[].at` names a known node; `spec.nodes[].via` (when present) names a vertex on that node. For the agent: `spec.agentVia` (when present) on the node resource names a vertex on that node. The `via` / `agentVia` field may be omitted when the target node has exactly one vertex; the compiler picks that vertex automatically. When the node has multiple vertices and the field is absent, validation fails with "specify `via` to disambiguate among `<list of vertex names>`." (In C0 every node has exactly one vertex, so these fields are always optional.)
13. **Access consistency** - every principal's role permits only groups defined by `FloreteGroup` resources; every service's group is defined; every service with an egress role has that role defined by a `FloreteRole` resource.
14. **Principal role coherence** - services that act as clients (outbound) must have a role declared; services without a role may only appear as targets.

Cert chain verification happens at runtime during mTLS, not at validate time — the repo doesn't hold individual certs.

## Compile Step

`florctl compile` produces **two artifacts per node** in C0 — `agent.json` (consumed by `flor agent run`) and `vertices/flor.json` (consumed by `flor vertex run`) — placed under `<repo>/.flor/compiled/<node>/mgmt/`. The per-node `mgmt/` subdirectory is the only output tree in C0; C1+ adds a sibling `ctrl/` (see [C1's ctrl-plane doc](../../../c1-manual-mesh/ctrl-plane/)). Both artifacts share a common envelope:

**Paths in compiled artifacts are scope-relative.** `ca_cert_path`, `cert_path`, and `priv_path` values are bare filenames (`ca.crt`, `alpha.crt`, …). The agent resolves them against the cluster install root (`clusters/<scope>/`) at startup. This keeps artifacts portable: the same compiled JSON is valid regardless of what the operator named the cluster scope locally, and a node joining multiple clusters can hold each cluster's artifacts in separate roots without any path collisions.

The install root is **flat**: the bundle delivers `ca.crt` (and the per-principal cert+key files) directly into `clusters/<scope>/`, regardless of how the repo organises them under `certs/`. The repo's `certs/management-planes/<name>.crt` shape is for human/audit organisation; the runtime sees only the flat install root, with mgmt-signer pubkeys delivered via the artifact's `authorized_mgmt_signers` field rather than as on-disk files.

```json
{
  "schema_version": "1.0",
  "plane": "mgmt",
  "artifactClass": "agent",
  "version": 42,
  "node": "alpha",
  "generated_at": "2026-04-20T12:00:00Z",
  "payload": { ... },
  "authorized_mgmt_signers": [
    { "spiffe_id": "spiffe://rete-lovers/management-plane/primary",
      "pubkey":    "<base64-ed25519-pubkey>" }
  ],
  "signature": {
    "alg": "ed25519",
    "key_id": "spiffe://rete-lovers/management-plane/primary",
    "value": "<base64>"
  }
}
```

Field semantics:

- `schema_version` — semver of the compiled-artifact public contract.
- `plane` — `"mgmt"` for everything in C0/C1. Reserved: `"ctrl"` for the unsigned control-plane decision stream that lands in B1+ (carried in a separate envelope, not inside this one). Discriminating now keeps the envelope schema stable across the split.
- `artifactClass` — `"agent"` for `agent.json` and `"vertex"` for vertex configs. This avoids overloading `kind`, which is reserved for source manifest resource kinds.
- `version` — **monotonic per-compilation number**. Lets future delta-based distribution ask "what version do you have?" without re-sending the full tree, and lets agents reject older-than-current artifacts (rollback-attack defense).
- `authorized_mgmt_signers` — the cluster's currently-authorized mgmt-envelope signer set, derived by the compiler from the `FloreteCluster` resource's `spec.signers.mgmt.keys` list. Each entry pairs the signer's SPIFFE URI with its ed25519 pubkey. Agents accept an artifact only if `signature.key_id` matches one of these entries and the signature verifies against the corresponding pubkey. The bootstrap snapshot of this list also ships in the enrollment bundle; thereafter the list lives in every fetched artifact, so signer rotation propagates via the normal `flor agent sync` flow (operator adds the new signer to the cluster manifest, publishes, agents sync, then operator switches to signing with the new key — standard overlap-window rotation).
- `signature` — operator signature over the canonical-form envelope minus the `signature` field itself. `key_id` is the SPIFFE URI of the **management-plane signing principal** (e.g. `spiffe://<cluster>/management-plane/primary`) — a sign-only identity, issued independently from any user entry via `florctl ca sign --kind management-plane`. See [ADR-0005](../../../../contributing/adr/0005-use-spiffe-identities) for the kind separation. The signing key never leaves operator hardware.

In B1+, a parallel `plane: "ctrl"` envelope (unsigned) will carry CP decisions and reference the mgmt `version` it obeys; agents will refuse a ctrl artifact whose claimed mgmt version is older than what the agent already holds. C0/C1 do not produce ctrl artifacts.

### Terminology

The payload uses two pairs of terms consistently, both from **flor's point of view** at the network boundary:

- **ingress** / **egress** — peer-to-local and local-to-peer QUIC sessions respectively. `ingress` lists which remote principals may connect to our local principals; `egress` lists which of our local principals may connect to a given remote target, along with how to dial that target.
- **inbound** / **outbound** (in `workloads[].io[].direction`) — the two ways a workload is wired into flor locally. `inbound` means the workload connects *into* flor here (flor accepts traffic and originates outgoing connections on its behalf — the workload acts as a **principal**). `outbound` means flor connects *out* to the workload here (flor delivers incoming connections — the workload acts as a **target**, i.e. a service). A workload can have both classes of `io` channels if it acts as principal and target. `bidirectional` is reserved for FlorIO between recursive flor layers (not used in C0).

Concretely in C0: the SOCKS5 listener flor exposes for a service's outbound calls is an `inbound` channel of `ioClass: socks5`; the loopback `127.0.0.1:port` upstream a service binds to is an `outbound` channel of `ioClass: tcp`. (These replace C0's earlier `socks5_proxy` / `upstream_addr` pair.)

### Example: agent.json

```json
{
  "schema_version": "1.0",
  "plane": "mgmt",
  "artifactClass": "agent",
  "version": 42,
  "node": "alice-laptop",
  "generated_at": "2026-04-20T12:00:00Z",
  "payload": {
    "ca_cert_path": "ca.crt",
    "vertices": [
      { "name": "flor", "config": "vertices/flor.json" }
    ],
    "control_plane": {
      "principal":     "spiffe://rete-lovers/node/alice-laptop",
      "config_server": "spiffe://rete-lovers/service/config-server"
    }
  },
  "authorized_mgmt_signers": [
    { "spiffe_id": "spiffe://rete-lovers/management-plane/primary",
      "pubkey":    "<base64-ed25519-pubkey>" }
  ],
  "signature": { "alg": "ed25519", "key_id": "spiffe://rete-lovers/management-plane/primary", "value": "<base64>" }
}
```

The agent reads its supervised vertices' configs to find the SOCKS5 port for its own `node/alice-laptop` principal — that's how it dials the config-server.

### Key points

- **One workloads list per vertex.** A workload is a **principal** if it has at least one `inbound` io channel, a **target** if it has at least one `outbound` io channel, both if both. The schema doesn't privilege users vs services — they're all just workloads with different SPIFFE-ID kinds and io shapes.
- **The agent's `node/<node>` principal is also a workload.** It lives in the vertex payload like any other principal, with a SOCKS5 inbound that the agent dials when it needs to talk to config-server. No special-casing in the runtime.
- **No forwarding table, no labels** — every tunnel in C0 is a direct QUIC connection between the initiator's vertex and the target service's vertex. The compiler pre-resolves only (a) which remote peers may initiate to this node's services (`ingress`) and (b) where this node's principals may initiate to (`egress`).
- **Egress is advisory, not security-enforcing.** The authoritative check happens at the target's `ingress`. `egress` exists so the initiator's vertex can dial the correct UDP address and fail fast on disallowed SOCKS5 requests. A compromised initiator could ignore its own egress table; it still cannot pass the target's ingress gate. Do not treat egress as an access-control boundary — it's a dialing table with a local convenience filter.
- **Roles expand at compile time.** `FloreteRole` manifests are source-only; the compiler resolves roles to explicit SPIFFE ID lists in `allow`. Role changes take effect on the next `florctl compile` + push — no cert reissuance needed, no role-resolution at runtime on the vertex hot path.
- **Deterministic** — same `florctl compile --repo X` produces identical output on any machine given identical inputs.
- **Per-node filtering** — each node's directory contains only the identity material and ACL rows relevant to its own local workloads.
